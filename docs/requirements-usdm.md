## 検証ステータス凡例

各仕様の先頭に付く3桁マーカーは、左から順に検証段階を表す。各桁は `■`（達成）/ `□`（未達成）で示す。

| マーカー | 意味 |
|---------|------|
| 1桁目 | 要件確定（レビュー済） |
| 2桁目 | 実装済（コードに反映済み・テスト未確認） |
| 3桁目 | テスト済（受け入れテストで検証済み） |

| 表記 | 状態 |
|------|------|
| `□□□` | 未着手 |
| `■□□` | レビュー済（実装未） |
| `■■□` | 実装済・テスト未 |
| `■■■` | テスト済 |

> 3桁目（`■■■`）への昇格は受け入れテスト実行時にのみ行う。

## 目的・スコープ

本文書は、Visual Studio Code向けマインドマップエディタ拡張機能（`vscode-mindmap-editor`）の要求仕様をUSDM形式で記述する。

本拡張機能は、Markdownファイルの見出し構造（H1〜H6）をインタラクティブなマインドマップとして可視化・編集する手段を提供する。すべての編集操作はMarkdownファイルに即時反映され、テキストエディタとの双方向同期を保証する。
## 用語定義

| 用語 | 定義 |
|------|------|
| ノード | マインドマップ上の一つの矩形要素。Markdownの見出し行に対応する |
| ルートノード | ツリーの頂点となるノード。H1見出しまたはファイル名 |
| 親ノード | 1つ以上の子ノードを持つノード |
| 子ノード | あるノードの直下に属するノード |
| 兄弟ノード | 同一親ノードに属し、同じ階層レベルを持つノード |
| 折りたたみ状態 | 子ノードを非表示にしているノードの状態 |
| フロントマター | Markdownファイル先頭の `---` で囲まれたYAMLブロック |
| 本文ノード | 見出しノードに紐付くリスト項目（チェックリスト・箇条書き）をツリー上に表示したノード |
## 動作環境

| 項目 | 要件 |
|------|------|
| VS Code バージョン | 1.85.0 以上 |
| 対象ファイル形式 | Markdown（`.md`） |
| Node.js（開発時） | 18.x 以上 |
| OS | Windows / macOS / Linux |

## ＜マインドマップ表示＞
### R-01 マインドマップ表示 #display

> **理由：** ユーザーがMarkdownファイルの見出し構造を視覚的にマインドマップとして把握・操作できるようにするため。

> **説明：** コマンドパレットの `Mind Map: マインドマップとして開く`（ショートカット: `Ctrl+Shift+M` / Mac: `Cmd+Shift+M`）でマインドマップパネルを開く。パネルはMarkdownエディタの横（`ViewColumn.Beside`）に表示される。ノードデザイン: 高さはテキスト折り返し有無で動的決定（1行:32px `NODE_H_1LINE`、2行:46px `NODE_H`）、幅はラベル長に応じて100〜500px動的決定（`NODE_MIN_W`〜`NODE_MAX_W`）、ボーダーを廃止し左端3pxのカラーアクセントバー（`::before`疑似要素）で階層識別。本文ノードは高さ1行30px（`BODY_H_1LINE`）・2行42px（`BODY_H`）動的決定、幅80〜450px（`BODY_MIN_W`〜`BODY_MAX_W`）動的決定。レイアウト間隔: 親右端→子左端 `H_GAP=20`、本文項目右端→ネスト子 `BODY_ITEM_GAP=12`、見出し兄弟縦間隔 `V_GAP=16`、本文項目縦間隔 `BODY_V_GAP=8`。
###### ＜見出し階層表示＞

- ■■■ R-01-01 `#`〜`######` の見出しをノードの階層として解釈して表示する。H1〜H6の6段階すべてが正しく階層化されること
- ■■■ R-01-02 ルートノードはH1見出しまたはファイル名（H1が存在しない場合）とする。H1があればH1をルートとし、なければファイル名をルートとすること
###### ＜ビュー切り替え・保存＞

- ■■□ R-01-03 MarkdownビューとMindMapビューを切り替えられる。エディタタイトルバーのボタン（またはコマンド）でマインドマップパネルを開閉できること
- ■■■ R-01-04 見出し間の本文（段落、コードブロック等）は構造変更時も保持される。移動・削除・追加後に見出し以外のテキストが失われないこと
- ■■■ R-01-07 ビューでの操作によりMarkdownに変更が反映されるたびに、自動でファイルを保存する。構造変更・名前変更・折りたたみ変更の反映後、VS Codeのダーティマーカーがなくなること
###### ＜テーマ・表示品質＞

- ■■□ R-01-05 マインドマップはVS Codeのカラーテーマ（ダーク/ライト）に追従する。VS Code標準のCSS変数を使用していること（推奨）
- ■■□ R-01-06 ノードのテキストが省略（…）されている場合、マウスオーバー時にツールチップで全文を表示する。テキストが省略されていないノードにはツールチップを表示しないこと
- ■■■ R-01-08 見出し直後に本文（body）が存在するノードに、インジケータードット（●）を表示する。ドットのホバーでbody内容がツールチップ表示されること。bodyが空のノードにはドットを表示しないこと（推奨）
###### ＜動的ノードサイズ＞

- ■■□ R-01-09 見出しノードの幅はラベルテキスト長に応じて動的に決定する。キャンバスのテキスト幅計測（`measureNodeW`）に基づき、最小100px（`NODE_MIN_W`）〜最大500px（`NODE_MAX_W`）の範囲で幅を算出し、各ノードへインラインスタイルとして適用すること。固定幅（旧260px）を用いないこと。高さはテキストの折り返し有無に応じて動的に決定する（`measureNodeH`）：1行に収まる場合は32px（`NODE_H_1LINE`）、折り返しが発生する場合は46px（`NODE_H`）。ラベルは最大2行までクランプ（`-webkit-line-clamp: 2`）表示すること。本文ノードは幅80〜450px（`BODY_MIN_W`〜`BODY_MAX_W`）、高さは1行30px（`BODY_H_1LINE`）・2行42px（`BODY_H`）で動的決定すること。幅計測はラベルのフォントサイズ（見出し13px・本文12px）でテキスト幅を測り、パディング・チェックボックス分のオーバーヘッドを加算して算出する。本文ノードは子項目を持つ（折りたたみトグル表示）場合のみトグルボタン幅 `TOGGLE_W=19px` を追加で加算する（`measureNodeW(text, isBody, hasToggle)`）

## ＜ノード操作＞
### R-02 ノード移動（ドラッグ＆ドロップ） #dnd

> **理由：** ユーザーがマインドマップ上でノードをドラッグして構造を直感的に組み替えられるようにするため。

> **説明：** ノード上でのドラッグはノード移動（D&D）として扱われ、背景パン（R-07-01）と競合しない。
###### ＜移動操作＞

- ■■□ R-02-01 同じ親ノード内での並び替えができる。ドロップ先ノードの上端25%で「前に挿入」、下端25%で「後に挿入」となること
- ■■□ R-02-02 別の親ノードへの移動（階層変更）ができる。ドロップ先ノードの中央50%で「子として追加」となること。レベルが自動更新されること
- ■■□ R-02-03 移動結果がMarkdownの見出し順・階層に即時反映される。ドロップ完了後、マークダウンファイルの内容が更新されること
###### ＜移動制限＞

- ■■□ R-02-04 ルートノードはドラッグできない。ルートノードのドラッグ操作を無視すること
- ■■□ R-02-05 ノードを自身の子孫ノードへドロップできない。循環参照が発生しないこと
- ■■□ R-02-08 H6ノードへの「子として追加」ドロップを禁止する。H6ノードの中央領域でドラッグ中は `not-allowed` カーソルを表示しドロップを無効化すること
###### ＜視覚フィードバック＞

- ■■□ R-02-06 ドラッグ中はゴーストプレビューを表示する。ドラッグ中にノードのテキストを示す半透明ゴーストが表示されること（推奨）
- ■■□ R-02-07 ドロップ位置を水平ラインで視覚的に示す。before/after はターゲットノードの上下辺に青ラインが表示され、inside はターゲットがハイライトされること
###### ＜ヒット判定＞

- ■■□ R-02-09 ドラッグ＆ドロップのヒット判定はノード矩形の左右いっぱい＋マージンで効く。ドロップ先判定の距離はノード中心ではなく矩形上の最近点までの距離（`distToRect`）で測り、ノード幅（動的サイズ `_w`）に追従して左右いっぱい＋許容マージン（`DROP_TOLERANCE=40`、レイアウト座標基準）まで判定が効くこと。中央付近のみに限定されないこと。縦方向の before/after/inside 判定（上端25%/下端25%/中央50%、R-02-01／R-02-02）は不変であること

### R-03 ノード編集（インライン編集） #edit

> **理由：** ユーザーがビューを切り替えずにマインドマップ上でノードのテキストを直接修正できるようにするため。

> **説明：** ルートノード（ファイル名）の保護は R-02-04（ドラッグ不可）・R-05-04（削除不可）と同列に扱う。`beginEdit` の単一チョークポイントで対象がルートの場合は編集を開始しない。
###### ＜編集操作＞

- ■■□ R-03-01 ノードをダブルクリックでインライン編集できる。ダブルクリックでテキスト入力フィールドがノード内に表示されること
- ■■□ R-03-02 編集内容がMarkdownの見出しテキストに即時反映される。Enterキーまたはフォーカスアウト時にMarkdownが更新されること
- ■■□ R-03-03 Escapeキーで編集をキャンセルできる。Escapeキー押下で変更を破棄し元のテキストに戻ること
- ■■□ R-03-04 空文字列への変更は無効とする。空文字でEnterを押した場合は変更を適用しないこと
###### ＜ルート保護＞

- ■■□ R-03-05 ルートノード（ファイル名）はインライン編集・リネーム不可とする。ルートノードのダブルクリック・`F2`・`Enter`（R-12-04 のルート編集フォールバック）いずれでもインライン編集が開始されないこと。子の見出しノード・本文ノードの編集は従来どおり可能であること

---
### R-04 ノード追加 #add

> **理由：** ユーザーがビューを切り替えずにマインドマップ上から新しいノードを追加して構造を拡張できるようにするため。
###### ＜追加操作＞

- ■■□ R-04-01 任意のノードに子ノードを追加できる。右クリックメニュー「子ノードを追加」で親より1段深い子が追加されること
- ■■□ R-04-02 同階層に兄弟ノードを追加できる。右クリックメニュー「兄弟ノードを追加」で同レベルのノードが直後に追加されること
- ■■□ R-04-03 追加したノードがMarkdownに見出しとして即時反映される。追加操作後にMarkdownファイルが更新されること
- ■■□ R-04-04 追加されたノードのデフォルトテキストは「新しいノード」とする。追加直後にインライン編集モードが自動的に開始されること

---
### R-05 ノード削除 #delete

> **理由：** ユーザーが不要な構造をビュー上から直接取り除けるようにするため。
###### ＜削除操作＞

- ■■□ R-05-01 ノードを選択してDeleteキーまたは右クリックメニューで削除できる。選択状態のノードにDeleteキーが有効なこと
- ■■□ R-05-02 子ノードが存在する場合は確認ダイアログを表示する。子を持つノード削除時に「子ノードをすべて削除しますか？」確認が表示されること
- ■■□ R-05-03 削除結果がMarkdownに即時反映される。削除確定後にMarkdownファイルが更新されること
###### ＜ルート保護＞

- ■■□ R-05-04 ルートノードは削除できない。ルートノードへの削除操作を無視すること

---
### R-08 ノード上下入れ替え #reorder

> **理由：** ユーザーがドラッグより素早く同一親ノードの子ノードの並び順をキーボードやメニューで調整できるようにするため。
###### ＜入れ替え操作＞

- ■■□ R-08-01 右クリックメニューの「↑ 上へ移動」「↓ 下へ移動」で兄弟間の順序を変更できる。選択ノードが1つ上/下の兄弟と入れ替わること
- ■■□ R-08-02 `Alt+↑` / `Alt+↓` ショートカットで選択中ノードを上下に移動できる。キーボードショートカットで順序変更が動作すること
- ■■□ R-08-03 先頭ノードの「上へ移動」、末尾ノードの「下へ移動」は無効化する。右クリックメニューの該当項目がグレーアウト（操作不可）になること
- ■■□ R-08-04 移動結果がMarkdownの見出し順に即時反映される。順序変更後にMarkdownファイルが更新されること

## ＜編集・同期＞
### R-06 折りたたみ #collapse

> **理由：** 大きな文書でも子階層を隠してノードを折りたたみ、全体像を把握しやすくするため。

> **説明：** フロントマター形式（`mindmap-collapse` キー）にルート（ファイル名）を除いた相対パス（「親/子」形式）を記録する。後方互換のため旧形式（ファイル名プレフィックス付き）パスも復元でき、保存時に新形式へ移行する。
###### ＜折りたたみ操作＞

- ■■■ R-06-01 子ノードを持つノードの折りたたみボタン（▼/▶）をクリックで展開/収縮できる。クリック後に子ノードの表示/非表示が切り替わること
- ■■□ R-06-05 ツールバーの「全展開」ボタンですべての見出しノード・本文ノード（子項目含む）を展開する。ルートノードを含むすべてのノードの折りたたみが解除され、本文項目の折りたたみ状態（`collapsedBodyItems`）もクリアされること。ツリーが存在しない場合は何もしないこと
- ■■□ R-06-06 ツールバーの「全折畳」ボタンですべての見出しノード・本文ノード（子項目含む）を折りたたむ。ルートノード自体は折りたたまず（折りたたむと何も表示されなくなるため）、ルート直下を含むすべての子孫ノードを折りたたむこと。子項目を持つすべての本文ノードも折りたたまれること。ツリーが存在しない場合は何もしないこと
- ■■□ R-06-07 折りたたみ状態のレイアウトへの反映は、左右展開レイアウト（`mindmap-left` 併用時）でも従来レイアウトと同等に行うこと。`root.collapsed` のときは左右どちらのレイアウト経路でもルートのみを配置し、ルート配下（左側・右側ノード）を描画しないこと（左右レイアウト経路が `root.collapsed` を無視してルート配下を展開したまま残す不具合の回避）
- ■■□ R-06-08 子見出しを持たず本文リスト項目のみを持つノードを折りたたんだ場合も、折りたたみパスをフロントマターに保存し、再パース時に復元すること。折りたたみパスを書き出す対象ノードの判定（子見出しあり または 本文項目あり）を Webview（`media/mindmap.js`）と Extension（`src/markdownParser.ts`）で一致させること
###### ＜状態永続化＞

- ■■■ R-06-02 折りたたみ状態をMarkdownのフロントマター（YAML）に保存する。折りたたみ操作後にフロントマターが更新されること
- ■■■ R-06-03 折りたたまれているノードのパスをリスト形式で記録する。フロントマターの `mindmap-collapse` キーに、ルート（ファイル名）を除いたルート直下からの相対パス（「親/子」形式）が保存されること。これによりファイルをリネームしても折りたたみ状態が保持される
- ■■■ R-06-04 次回ファイルを開いたとき、フロントマターを読み取り折りたたみ状態を復元する。ファイルを再度開いた際に折りたたみ状態が復元されること。後方互換のため、旧形式（ファイル名プレフィックス付き「ルート/親/子」）のパスも復元でき、一度保存すると新形式（相対パス）へ移行する
- ■■□ R-06-09 折りたたみ状態の保存（`saveCollapseState`）は管理フロントマターのみの書き込みであり、最新パース済みキャッシュツリーから再生成されるため、コンフリクト誤検知でこの書き込みが破棄されてはならない。`applyDocumentEdit` でコンフリクト検知をスキップして書き込み、`baseText` を確実に更新してから再同期すること（NF-03 のユーザー本文・他フロントマター非破壊は維持する）

### R-07 パン・ズーム #pan-zoom

> **理由：** 大きな構造でも全体を把握しながら作業できるよう、ビューを移動・拡大縮小できるようにするため。

> **説明：** ノード上でのドラッグはノード移動（D&D）として扱われ、パンとは競合しない。接続線（ベジェ曲線）上のドラッグもパンとして機能する。
###### ＜パン・ズーム操作＞

- ■■□ R-07-01 背景ドラッグでパン（移動）できる。ノード以外の領域（背景・接続線を含む）をドラッグしてビューが移動すること
- ■■□ R-07-02 マウスホイールでズームできる。ホイール操作でカーソル位置を中心にズームすること
- ■■□ R-07-03 ツールバーの「フィット」ボタンでビュー全体を画面に収め、ルートノードを画面中央に配置する。ツリー全体が画面内に収まる倍率に自動調整され、ルートノードの中心がビューポートの中心に一致すること
- ■■□ R-07-04 ショートカットキー `F` でフィット、`Ctrl++/Ctrl+-` でズームできる。キーボードショートカットが動作すること（推奨）
- ■■□ R-07-05 ルートノードは通常表示時と選択（フォーカス）時で視覚的に区別できる。通常時はノード色の外周枠で階層アクセントを表示し、選択時はそれと識別できる選択強調を重ねて表示すること。他ノードのアクセントバー配置（左ノードは右端）は維持すること

### R-09 ファイル保存（Ctrl+S） #save

> **理由：** 他のVS Codeエディタと同じ操作感で、マインドマップビューにフォーカスしたままMarkdownファイルを保存できるようにするため。
###### ＜保存操作＞

- ■■□ R-09-01 マインドマップビュー上で `Ctrl+S`（Mac: `Cmd+S`）を押すと対応するMarkdownファイルを保存する。キー押下後、VS CodeのタブからダーティマーカーがなくなりMarkdownが保存されること
- ■■□ R-09-02 保存はマインドマップ経由の最新編集内容を含む。直前のマインドマップ操作が反映された状態でファイルが保存されること
###### ＜保存インジケーター＞

- ■■□ R-09-03 保存完了時（自動保存・手動保存いずれも）、ツールバーに「✓ 保存済」インジケーターを約1.8秒表示する。保存のたびにフェードイン→フェードアウトすること。連続保存時はタイマーがリセットされること（推奨）

### R-10 Undo #undo

> **理由：** 誤操作を素早く取り消せるよう、マインドマップ上の操作を `Ctrl+Z` で戻せるようにするため。
###### ＜Undoスタック管理＞

- ■■□ R-10-01 Webview内に最大50操作のUndoスタックを保持する。51操作目以降は最古のエントリーが破棄されること
- ■■□ R-10-02 各変更操作（D&D・名前変更・追加・削除・上下移動・折りたたみ変更）の直前にツリースナップショットをUndoスタックに積む。1回の `Ctrl+Z` で1操作分だけ戻ること
- ■■□ R-10-03 `Ctrl+Z` でスタックから直前のスナップショットを取り出し、ツリーを復元してMarkdownに同期する。Undo後にMarkdownファイルの内容がスナップショット時点の状態になること
- ■■□ R-10-04 スナップショットにはノードのレイアウト座標（`_x`, `_y`, `_sh`）を含めない。`id / text / level / collapsed / body / children` の論理データのみを保存すること
###### ＜Undo後の状態＞

- ■■□ R-10-05 Undo操作後は選択状態をリセットする。Undo実行後に選択ノードが存在しない状態になること（推奨）

### R-11 双方向同期の競合管理 #conflict

> **理由：** マインドマップ操作中に別タブでMarkdownを直接編集した場合でも正しい状態を保ち、複数人の同時編集で変更が黙って上書き消去されないようにするため。

> **説明：** 競合発生時はMarkdownの内容を正として扱う。操作中（`isOperating = true`）はMarkdown変更イベントを無視するが、その間 base は更新されないため、操作完了後の全文置換前に base 比較に乗り外部変更が検知される。
###### ＜操作中の競合防止＞

- ■■□ R-11-01 構造変更・名前変更操作中（`isOperating = true`）はMarkdown変更イベントを無視する。D&Dやインライン編集の確定処理中にMarkdown変更イベントが割り込んでもマインドマップが中断されないこと
- ■■□ R-11-02 操作完了後、MarkdownをソースオブトゥルースとしてマインドマップをReparse・再描画する。操作完了後、Markdownと同期した最新の状態がマインドマップに反映されること
- ■■□ R-11-03 競合発生時はMarkdownの内容を正として扱う。同時編集が発生した場合、Markdownエディタ側の変更が最終的にマインドマップに反映されること
###### ＜楽観的同時実行制御（Lost Update防止）＞

- ■■■ R-11-04 マインドマップのキャッシュツリーが「どのドキュメント内容（テキスト）からパースされたか」を base スナップショットとして記録する。`syncFromDocument` でツリーを更新するたびに、その時点のドキュメントテキスト（改行は LF に正規化）が base として保持されること。base は書き込み成功後にも書き込んだ内容で更新されること
- ■■■ R-11-05 全文置換による書き込みの直前に、ライブのドキュメント内容およびディスク実体を base と比較してコンフリクトを検知する。base と一致する場合のみ従来どおり書き込み＋保存する。base と不一致（他者が並行変更）の場合は黙って上書きしないこと。改行コードのみの差異（CRLF/LF）はコンフリクトとして扱わないこと。自分の書き込みエコー（書き込もうとしている内容と現状が一致）はコンフリクトとして扱わないこと。共有ドライブで `TextDocument` がディスクより古い場合に備え、`vscode.workspace.fs.readFile` によるディスク読み取りも判定に併用すること（自動テスト `conflictDetection.test.ts` で判定ロジックを検証）
- ■■□ R-11-06 コンフリクト検知時はモーダル警告（`showWarningMessage`）でユーザーに解決方法を選択させる。「最新を読み込む（自分の編集は破棄）」選択時は他者の変更を保持してディスクから再同期し、自分の書き込みを中止すること。「自分の変更で上書き（他者の変更は破棄）」選択時のみ従来どおり全文置換すること。ダイアログを閉じる（未選択）場合は安全側（最新を読み込む）として扱うこと
- ■■□ R-11-07 コンフリクト解決で破棄される側の内容を、衝突ファイルとしてバックアップ退避する。上書き選択時は他者の版を、最新読み込み選択時は自分の版を、`<ファイル名>.conflict-<mine|remote>-<timestamp>.md` として同じディレクトリに保存すること。バックアップ失敗はコンフリクト解決をブロックしないこと（ベストエフォート）（推奨）
- ■■■ R-11-08 操作中（`isOperating = true`）に届いた外部変更も、操作完了後の書き込み前検知で必ずコンフリクトとして扱う。操作中は外部変更イベントを無視する（R-11-01）が、その間 base は更新されないため、操作完了後の全文置換前に base 比較に乗り、外部変更が検知・分岐されること。既存の `_editQueue` 直列化・echo ループガード（`applyingEdit`）が壊れないこと

### R-19 アクティブエディタへの自動追従 #follow

> **理由：** ユーザーがビューアをいちいち開き直さずに、複数のMarkdownファイルを行き来しながら構造を確認・編集できるようにするため。

> **説明：** 実装は Extension Host 側（`extension.ts` の `onDidChangeActiveTextEditor` リスナー → `MindMapPanel.followActiveDocument`）。パネルの差し替えは `MindMapPanel.switchDocument` / `performSwitch` が担う。`switchDocument` は `_editQueue` にチェーンして実行し、旧ドキュメントへの保留中書き込みの完了後に状態を入れ替える。パネルを閉じた場合、追従対象（`activePanel`）は他の開いているパネルへフォールバックする。
###### ＜追従動作＞

- ■■□ R-19-01 アクティブなテキストエディタが別のMarkdown（`languageId === 'markdown'`）ドキュメントに切り替わったら、開いているマインドマップビューアの対象ドキュメントをそのドキュメントへ差し替えて再描画する。別の `.md` ファイルをエディタで選択・表示すると、ビューアの表示内容・タイトルが新しいファイルに追従すること（`vscode.window.onDidChangeActiveTextEditor` を使用）
- ■■□ R-19-02 非Markdown（出力パネル・設定画面・画像など `languageId !== 'markdown'`）にフォーカスが移った場合は、ビューアの内容を維持する（クリア・破棄しない）。非Markdownへフォーカスが移ってもビューアが直前のMarkdownを表示し続けること
- ■■□ R-19-03 追従対象は「アクティブなマインドマップパネル」とし、同一ドキュメントに対して二重のパネルを生成しない。既に対象ドキュメントを表示しているパネルがある場合はそれをアクティブ扱いとし、新規パネルを作らないこと。複数パネルがある場合は最後に作成／フォーカスしたパネルが追従対象になること
- ■■□ R-19-04 ドキュメント差し替え時に、双方向同期・コンフリクト検知/解決の状態（`baseText` / `_editQueue` / `isOperating` / `applyingEdit` / キャッシュツリー・折りたたみパス・チェックボックスマイグレーション）を新ドキュメント基準にリセットし、矛盾を生じさせない。切替前ドキュメントへの書き込み（`_editQueue`）が完了してから差し替えること。差し替え後は新ドキュメントを新規オープン同様に扱い、外部変更同期リスナー（`onDidChangeTextDocument`）も新ドキュメントへ再バインドすること。ユーザーのMarkdown本文・フロントマターを書き換えない（NF-03）
###### ＜設定＞

- ■■□ R-19-05 自動追従は設定 `mindmap.followActiveEditor`（既定 `true`）でON/OFFできる。`false` のときはエディタを切り替えてもビューアが追従しないこと（推奨）

### R-12 キーボードナビゲーション #keyboard

> **理由：** ユーザーがマウスに頼らずキーボードだけでマインドマップの素早い構造把握・編集を完結できるようにするため。

> **説明：** ショートカット一覧: `↑/↓`（前後の可視兄弟ノードまたは本文項目を選択）、`→`（子方向へ移動、折りたたみ中なら展開）、`←`（折りたたみ、または親ノードへ移動）、`Enter`（兄弟ノードを直後に追加して編集開始）、`F2`（インライン編集を開始）、`Tab`（子ノードを追加して編集開始）、`Escape`（選択を解除）、`Delete`（選択ノードを削除）、`Alt+↑/↓`（兄弟間の上下入れ替え）、`Ctrl+Z`（直前の操作を取り消す）。左右展開レイアウト（R-20）を含め、配置に関係なく `→` は常に子方向、`←` は常に折りたたみ/親方向として動作する。
###### ＜ナビゲーション＞

- ■■□ R-12-01 `↑/↓` キーで見出しノードの可視兄弟間を移動できる。移動対象は同一親ノードの `children` に含まれる同階層の可視兄弟ノードだけとし、見出しと本文項目の境界を跨がないこと。ルートノードは兄弟を持たないため `↑/↓` では何もしないこと
- ■■□ R-12-02 `→` キーで子方向へ移動する。左右配置（`direction='left'` を含む）に関係なく常に最初の子見出しノードを優先して選択し、子見出しがない場合は最初の本文ノードを選択すること。折りたたみ中なら展開してから移動すること。子見出し・本文ノードのいずれもない場合は何もしないこと
- ■■□ R-12-03 `←` キーで折りたたみ/親方向へ移動する。左右配置に関係なく、ルート以外では子見出しまたは本文ノードを持つ展開中ノードを先に折りたたみ、すでに折りたたみ済み（または子なし）の場合は親ノードへ移動すること。ルートノード選択中は折りたたまず、`root.children` のうち `side === 'left'` の先頭可視直下ノードへ移動し、左ノードがなければ何もしないこと
- ■■□ R-12-08 キーボード選択変更時に対象ノードが画面外にある場合、ビューをスクロールして表示する。ノードが60px以上画面端に近い場合に自動パンすること
- ■■□ R-12-11 本文ノード（bodyItem）選択中の矢印キーナビゲーション: `↑/↓` は同じ直接親を持つ可視兄弟本文項目間だけを移動対象とし、トップレベル本文項目（`indent === 0`）では同じ見出し配下のトップレベル本文項目間、ネスト本文項目（`indent > 0`）では同じ親本文項目の `children` 間だけを移動すること。見出しと本文項目の境界を跨がないこと。兄弟境界では何もしないこと。`→` キーは左右配置に関係なく常に子方向とし、現在の本文ノードの最初の子本文ノードへ移動すること。折りたたみ中なら展開してから移動し、子がない場合は何もしないこと。`←` キーは左右配置に関係なく常に折りたたみ/親方向とし、子本文ノードを持つ展開中の本文ノードでは先にその本文ノードを折りたたみ、そうでない場合は `indent === 0` なら親の見出しノードへ、`indent > 0` なら親の本文ノードへ移動すること
###### ＜編集ショートカット＞

- ■■□ R-12-04 `Enter` キーで選択中ノードの直後に兄弟ノードを追加し、直ちにインライン編集を開始する。ルートノード（親なし）の場合は兄弟追加を行わない。ルートはインライン編集不可（R-03-05）のため、フォールバックの編集開始も行われず何も起きないこと
- ■■□ R-12-04b `F2` キーまたはダブルクリックで選択中ノードのインライン編集を開始する。選択ノードにテキスト入力フィールドが表示されること
- ■■□ R-12-05 `Tab` キーで選択中ノードに子ノードを追加し、直ちにインライン編集を開始する。H6ノードでは `Tab` を無効化すること
- ■■□ R-12-06 `Escape` キー（非編集中）で選択を解除する。選択ハイライトが消えること
- ■■□ R-12-07 `Delete` キーで選択中ノードを削除する（既存 R-05-01 のキーボード操作と統合）。子を持つ場合は確認ダイアログを表示すること
- ■■□ R-12-10 `Ctrl+Z` でUndoスタックから直前の操作を取り消す（US-10 参照）。Undoスタックが空の場合は何もしないこと
###### ＜編集中の競合防止＞

- ■■□ R-12-09 編集中（`editingId` が設定中）はナビゲーション系ショートカットを無効化する。入力フィールド内での操作がナビゲーションと競合しないこと

### R-13 本文項目のマインドマップツリー表示 #body-item

> **理由：** ユーザーがノードに紐付くリスト項目（チェックリスト・箇条書き）をタスクや補足情報として見出しと同じビューで管理・操作できるようにするため。

> **説明：** 本文ノードの高さは 42px（`BODY_H`・2行表示対応。見出しノード 46px よりわずかに小さい）。ラベルは最大2行までクランプ（`-webkit-line-clamp: 2`）表示する。本文ノードの幅はラベル長に応じて動的に決定する（最小80px=`BODY_MIN_W`〜最大450px=`BODY_MAX_W`）。本文ノードはネスト（子本文ノード）を持てる（R-15 参照）。本文行の変更は `editBody` メッセージで拡張機能に通知する。本文ノードのレイアウト順：見出し子ノードが上、本文ノードが下。
###### ＜本文ノード表示＞

- ■■□ R-13-01 本文中の `- [ ] item` / `- [x] item` / `- item` を、見出しノードの右に子要素として表示する（本文ノード）。本文ノードは見出しノードと明確に異なるデザイン（破線ボーダー・半透明背景・小フォント）で表示されること。本文ノードの幅はラベル長に応じて動的に決定され、子項目を持つ（折りたたみトグル ▼/▶ が表示される）本文ノードではトグルボタンの占有幅も加味してテキスト幅を確保し、短いテキストが不要に折り返されないこと
- ■■□ R-13-02 本文ノードの接続線を破線（dashed）・グレー色にして見出し間の実線と区別する。見出しノード→見出しノードの接続線は変更しないこと
- ■■□ R-13-04 `- item`（チェックボックスなし）行はダッシュ（–）付きで表示する。チェックボックス行と視覚的に区別できること（推奨）
- ■■□ R-13-08 非リスト形式の本文行（段落・コードブロック等）はツリーに表示せず、見出しノード上のドット（●）インジケーターで存在を示す。ドットのホバーで本文全文がツールチップ表示されること（推奨）
###### ＜チェックボックス操作＞

- ■■□ R-13-03 `- [ ] item` / `- [x] item` 行はチェックボックス付きで表示し、クリックで即座にチェック状態を切り替えてMarkdownに保存する。`- [x]` 行はラベルに取り消し線を表示し、ノード全体を半透明にすること
###### ＜本文ノード編集・削除＞

- ■■□ R-13-05 本文ノードをダブルクリックまたは F2 でインライン編集できる。入力はテキストのみとし、`- [ ] ` プレフィックスは自動付与される。編集確定後に Markdown の該当行が更新されること。入力欄にチェックボックス記号を入力させないこと
- ■■□ R-13-06 本文ノードを選択した状態で Delete キーを押すと、その本文行を削除する。削除後に Markdown ファイルが更新されること
- ■■□ R-13-07 見出しの折りたたみボタン（▼/▶）は、本文ノードが存在する場合にも表示される。折りたたむと本文ノードも非表示になること
###### ＜本文ノード追加・移動＞

- ■■□ R-13-09 本文ノードを選択した状態で Enter キーを押すと、同一親・同一インデントレベルの兄弟本文項目をサブツリー末尾の直後に追加し、自動編集モードに入る。追加直後にインライン編集が開始されること。右クリックメニュー「同階層に追加」でも同等の操作が可能なこと
- ■■□ R-13-10 本文ノードはドラッグ&ドロップで移動できる。同一親内での並び替え、別の見出しノードまたは別親の本文ノード前後へのドロップが可能なこと。ドロップ位置は水平ラインで視覚的に示すこと。ドラッグ時は子項目も一緒に移動されること。移動後の階層（インデント）に応じて、トップレベル（indent=0）はチェックボックス形式、ネスト（indent>0）はダッシュ形式に自動変換されること。ドロップ先判定のヒット範囲は本文ノード矩形の左右いっぱい＋マージンで効き（R-02-09 と同方式）、中央付近のみに限定されないこと
###### ＜チェックボックスマイグレーション＞

- ■■■ R-13-11 マインドマップビューを開いた時、チェックボックス形式になっていないトップレベル（indent=0）の本文項目（プレーン箇条書き `- text`）に、空のチェックボックス（`- [ ] `）を自動付与して正規化（マイグレーション）し、変更があればファイルに保存してから編集可能な状態にする。パネル初期化（Webview の `ready` 受信時、または `ready` 不達時のフォールバック）でトップレベルのプレーン箇条書き行を `- [ ] text` へ変換し、変更があった場合のみ既存の保存経路（`applyDocumentEdit` / `_editQueue`）で書き戻すこと。変換が不要（既にすべてチェックボックス／対象行なし）な場合は書き込みを行わずファイルをダーティにしないこと。マイグレーションは1パネルにつき1回のみ実行すること。既存のチェックボックス（`- [ ]` / `- [x]` / `- [X]`）はチェック状態を保持してそのまま、ネスト（indent>0）のダッシュ項目はダッシュのまま、非リスト行（段落・コードブロック・通常文）は変更しないこと（NF-03 厳守）。コードフェンス（``` / ~~~）内のリスト風の行も変換対象外とすること。純粋ロジックは `src/bodyItems.ts`（`normalizeBodyCheckboxes` / `normalizeTreeCheckboxes`）に実装し自動テストで検証する（`media/mindmap.js` にもミラー）

### R-15 本文項目の階層構造（ネスト） #body-nest

> **理由：** ユーザーがタスクのサブタスクや補足情報をMarkdownの箇条書きインデントに準拠した階層で管理し、マインドマップツリー上でも階層関係を視覚的に確認できるようにするため。

> **説明：** 子本文項目は親本文項目の実幅 + `BODY_ITEM_GAP`（12px）の水平位置に配置される（旧 `BODY_H_SPACE` 固定値 220px は廃止し、動的サイズに合わせた配置へ変更済み）。既存Markdownに `- [x] ` 形式の子項目がある場合も子項目扱い（ただし表示はチェックボックスなし）。子本文項目のさらに下の階層も同様のルールで再帰的にサポートする。
###### ＜ネスト表示＞

- ■■□ R-15-01 本文項目はMarkdownのインデント（2スペース単位）に準拠した階層構造を持てる。親項目より2スペース以上多いインデントの行は子項目としてツリーに表示されること。子本文ノードは親本文ノードの実幅（`item._w`・動的）に固定ギャップ `BODY_ITEM_GAP=12px` を加えた位置（`x + item._w + BODY_ITEM_GAP`）に配置し、親子間の重なりを防ぐこと。子項目を持つ親本文ノードの実幅は、折りたたみトグルボタンの占有幅（`TOGGLE_W=19px`）をオーバーヘッドに加算して算出し、トグル表示時もテキストが折り返されないこと
- ■■□ R-15-02 子階層（インデントあり）の本文項目はチェックボックスを持たず、ダッシュ記号（–）で表示する。子本文項目を新規追加した場合も `- text`（チェックボックスなし）形式で保存されること
- ■■□ R-15-06 本文項目→子本文項目の接続線は、見出し→本文項目と同じ破線グレースタイルで表示する。見出し間の実線と視覚的に区別できること（推奨）
###### ＜ネスト操作＞

- ■■□ R-15-03 Tab キーで選択中の本文項目に子項目を追加し、インライン編集を開始する。挿入される行のインデントが親より2スペース多いこと。追加直後にインライン編集が開始されること
- ■■□ R-15-04 右クリックメニュー「子項目を追加」で子本文項目を追加できる。Tab キーと同等の動作をすること
###### ＜ネスト折りたたみ＞

- ■■■ R-15-05 本文項目に子項目が存在する場合、折りたたみボタン（▼/▶）を表示し、子項目を折りたたみ可能にする。折りたたむと子項目が非表示になること。折りたたみ状態はMarkdownファイルのフロントマター（`body-item-collapse:` キー）に `見出し相対パス::項目チェーン` 形式（見出し部はルート＝ファイル名を除いた相対パス）で保存し、ファイルを閉じて再度開いた際も復元されること。後方互換のため旧形式（ファイル名プレフィックス付き見出しパス）も復元でき、保存時に新形式へ移行する

## ＜レイアウト＞
### R-20 左右展開レイアウト #layout-side

> **理由：** ユーザーがルート直下の H1 ノードを左右両方向に展開することで、双方向マインドマップとして情報を整理・比較できるようにするため。

> **説明：** `MindMapNode` の `side?: 'left' | 'right'` フィールドでルート直下 H1 ノードの展開方向を管理する。H2 以下は親の direction を継承して再帰配置される。左側配置時、ノードとその配下はルートの左方向に展開され、接続線もルート左端 → 子右端の方向で引かれる。フロントマター `mindmap-left` キー（`mindmap-collapse` と同じパターン）で永続化する。
###### ＜左右レイアウト＞

- ■■□ S20-01 `mindmap-left` フロントマターキーに記載された H1 ノードのテキストに一致するルート直下ノードに `side: 'left'` が付与される。含まれないルート直下ノードは `side: 'right'`（デフォルト）になること
- ■■□ S20-02 `mindmap-left` がないファイルではすべてのルート直下ノードが `side: 'right'` になること
- ■■□ S20-03 左側ノード（`side: 'left'`）が存在する場合、ルートノードはキャンバス左端から左側サブツリー幅分オフセットした位置に配置される。左側ノードが存在しない場合は従来どおり `x = PAD` に配置されること
- ■■□ S20-04 左側 H1 ノードとその配下（H2 以下）は `direction = 'left'` で配置され、各子ノードが親の左端 - `H_GAP` - 子幅の x 座標に配置されること
- ■■□ S20-05 接続線（Bezier 曲線）は `direction = 'left'` のとき親の左端から子の右端へ引かれる。右方向ノードは従来どおり親の右端から子の左端へ引かれること
- ■■□ S20-06 左側ノードが存在する場合、キャンバス全体の最小 x 座標が `PAD` 未満にならないよう全ノードを右シフトする（`shiftTree`）こと
- ■■□ S20-14 左右レイアウト経路（左側ノードが存在する分岐）でも `root.collapsed` を尊重し、ルート折りたたみ時はルートのみ配置して左右の子を配置しないこと（従来レイアウト経路の `assignPositions` 早期 return と同等の挙動）
###### ＜ドラッグ&ドロップで side 変更＞

- ■■□ S20-07 H1 ノードをドラッグしてルートノードの左半分（中心線より左）でドロップすると `side = 'left'` に設定され、左側レイアウトで配置されること
- ■■□ S20-08 H1 ノードをドラッグしてルートノードの右半分（中心線より右）でドロップすると `side = 'right'` に設定され、右側レイアウトで配置されること
- ■■□ S20-09 ドラッグ中にルートの左ゾーン・右ゾーンをそれぞれ `drop-root-left` / `drop-root-right` クラスでハイライト表示すること
- ■■□ S20-10 H2 以下のノードをルートにドロップした場合も H1 に昇格（level = 1）してから side が設定されること
###### ＜永続化＞

- ■■□ S20-11 `side = 'left'` のルート直下ノードのテキストをフロントマター `mindmap-left` キーに相対パス形式で保存・復元すること（`mindmap-collapse` と同じパターン。ファイルリネームに対して安全）
- ■■□ S20-12 左側ノードが存在しなくなった場合、`mindmap-left` ブロックをフロントマターから削除すること
- ■■□ S20-13 `setSide` メッセージ（Webview → Extension）でルート直下ノードの side を更新し、フロントマターに反映して保存すること

## ＜高度機能＞
### R-16 チェックボックス進捗ウィジェット #progress

> **理由：** ユーザーがドキュメント全体のチェックボックス完了状況を一目で把握しながらタスク管理できるようにするため。

> **説明：** 集計対象はツリー全体のすべての見出しノードの `body` フィールド内のチェックボックス行。ウィジェットはツールバーの折畳ボタン群の右に配置する（常時ヒントテキストは廃止）。
###### ＜進捗表示＞

- ■■□ R-16-01 ツールバーに全ノードのチェックボックス総数と完了数を「✓ 完了数 / 総数」形式で表示する。チェックボックスが1つもない場合は非表示にすること。チェック状態の変更後に即時更新されること。4桁の数値（最大9999件）まで見切れずに表示できる最小幅（92px）を確保すること
- ■■□ R-16-02 進捗ウィジェットの背景にグリーンの進捗バーを半透明で表示する。進捗率（完了数/総数）に応じてバー幅が変化すること。状態変化時にアニメーションすること（推奨）
- ■■□ R-16-03 全チェックボックスが完了した場合、ウィジェットの枠線をグリーンに変化させる。全完了→未完了に戻った際も枠線が元の色に戻ること（推奨）
###### ＜ショートカットヘルプ＞

- ■■□ R-16-04 キーボードショートカット一覧はツールバー常時表示をやめ、「?」ボタンのクリックでポップアップとして表示する。ポップアップはツールバー右端に固定表示され、ボタン再クリック・領域外クリックで閉じること。ショートカット一覧は見出しノード・本文項目・共通の3セクションに分類して表示すること

### R-17 コピー・カット＆ペースト #clipboard

> **理由：** ユーザーが選択中のノードまたは本文項目を複製・移動・再利用して構造を素早く組み立てられるようにするため。
###### ＜見出しノードのコピー・ペースト＞

- ■■□ R-17-01 `Ctrl+C`（Mac: `Cmd+C`）で選択中の見出しノードを内部クリップボードにコピーする。子ノード・本文を含む全ツリー構造がコピーされること
- ■■□ R-17-02 `Ctrl+V`（Mac: `Cmd+V`）で見出しノードを選択中ノードの子として貼り付ける。選択ノードの子階層末尾に追加されること。貼り付け先の子レベルに合わせてレベルが自動調整されること。新しいIDが付番されること。選択ノードがH6の場合は貼り付けを行わないこと
- ■■□ R-17-07 `Ctrl+X`（Mac: `Cmd+X`）で選択中の見出しノードをカットする（コピー後に元ノードを削除）。カット後に元のノードがツリーから削除されること
###### ＜本文項目のコピー・ペースト＞

- ■■□ R-17-03 `Ctrl+C` で選択中の本文項目を子項目ごとコピーする。子項目（ネスト構造）も含めて全行がコピーされること
- ■■□ R-17-04 `Ctrl+V` で本文項目を選択中本文項目のサブツリー末尾の直後に貼り付ける。貼り付け先の階層に応じてチェックボックス/ダッシュ形式が自動調整されること
- ■■□ R-17-05 `Ctrl+V` で本文項目クリップボードを見出しノード選択中に貼り付けるとトップレベル本文として追加される。チェックボックス形式（`- [ ] text`）で末尾に追加されること
- ■■□ R-17-08 `Ctrl+X` で選択中の本文項目をカットする（コピー後に元行を削除）。カット後に元の本文行がMarkdownから削除されること
###### ＜編集中の操作制限＞

- ■■□ R-17-06 インライン編集中（`editingId` が設定中）は Ctrl+C / Ctrl+X / Ctrl+V のマインドマップ操作を無効化する。入力フィールド内でのテキストコピーペーストが正常に動作すること

### R-18 複数選択 #multi-select

> **理由：** ユーザーが複数のノードや本文項目を一括で移動・コピー・カット・削除して効率よく操作できるようにするため。
###### ＜複数選択操作＞

- ■■□ R-18-01 `Ctrl+クリック`（Mac: `Cmd+クリック`）で見出しノードを複数選択できる。既に選択中のノードをCtrl+クリックすると選択解除されること。異なる階層のノードも選択可能なこと
- ■■□ R-18-02 `Ctrl+クリック` で本文項目を複数選択できる。既に選択中の本文項目をCtrl+クリックすると選択解除されること
###### ＜複数選択での操作＞

- ■■□ R-18-03 複数選択状態のノードをドラッグ＆ドロップで移動できる。選択中のいずれかのノードをドラッグすると、複数ノードがまとめて移動されること。移動先のターゲットノードの子孫である選択ノードはドロップ対象から除外されること
- ■■□ R-18-04 複数選択状態で `Ctrl+C` を押すと、選択中の全ノードをコピーする。ペースト先ノードの子階層にまとめて追加されること
- ■■□ R-18-05 複数選択状態で `Ctrl+X` を押すと、選択中の全ノードをカットする。カット後に元のノードがすべてツリーから削除されること
- ■■□ R-18-06 複数選択状態で `Delete` を押すと、選択中の全ノードを削除する。子ノードを持つノードが含まれる場合は確認ダイアログを表示すること
###### ＜複数選択の解除＞

- ■■□ R-18-07 右クリック・矢印キー・F2などの単一操作キーは複数選択を解除して単一選択として動作する。右クリック時・キーナビゲーション時に複数選択状態がリセットされること
- ■■□ R-18-08 `Escape` キーで複数選択状態を解除する。複数選択がすべてクリアされること
