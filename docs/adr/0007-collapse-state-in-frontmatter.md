# ADR-0007: 折りたたみ状態を YAML フロントマターに永続化

- ステータス: 承認済み
- 確信度: 中（設計の事実は明確だが選定理由の明示はコード内コメントのみ）
- 日付: v1.8.4（commit c776d2e）で本文子ノード分も含め確立

## コンテキスト

マインドマップ上でノードや本文項目を折りたたんだ状態を、VS Code を再起動しても・ファイルを別のマシンで開いても維持したいという要件（NF-03 参照）があった。状態をどこに保存するかという設計判断が必要だった。

## 決定

折りたたみ状態を Markdown ファイル自身の YAML フロントマターに書き込む。

```yaml
---
mindmap-collapse:
  - "ノードA/子ノードB"
body-item-collapse:
  - "ノードA::本文項目チェーン"
---
```

パスはファイル名をプレフィックスとせず**ルート相対**で保存し、ファイルリネーム時にパスが壊れない設計にした（`markdownSerializer.ts:14`、`markdownParser.ts:82-84`）。

## 理由

- ファイルと状態が一体化するため、ファイルをコピー・共有・Git 管理しても状態が追従する。
- VS Code の `globalState` や `workspaceState` に保存すると、別マシン・別ワークスペースで開いたときに状態が引き継がれない。
- サイドカーファイル（`.mindmap-state.json` 等）は管理ファイルが増える（`.gitignore` 設定なども必要）。

パスをルート相対で保存する理由：ファイル名が変わると絶対パスが壊れるため（`markdownSerializer.ts` コメント参照）。後方互換のためファイル名プレフィックスつきの旧形式も読めるようにしてある。

## 捨てた選択肢

- **VS Code `globalState` / `workspaceState`**: ファイルをポータブルに扱えない（別マシンで状態が失われる）。
- **サイドカーファイル**: ファイルが増える、`.gitignore` 設定が必要になる可能性。

**Why:** (要確認) 「ファイル横断で永続化」というコミットメッセージから、別ファイル方式への不満が動機と推測。
