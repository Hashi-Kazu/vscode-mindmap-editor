# マインドマップエディタ VS Code拡張機能 要求仕様書

**文書番号**: MME-REQ-001  
**バージョン**: 2.4.0  
**作成日**: 2026-06-04  
**最終更新**: 2026-06-17  
**ステータス**: 承認済み

---
## 1. 目的・スコープ

本文書は、Visual Studio Code向けマインドマップエディタ拡張機能（`vscode-mindmap-editor`）の要求仕様を定義する。

本拡張機能は、Markdownファイルの見出し構造（H1〜H6）をインタラクティブなマインドマップとして可視化・編集する手段を提供する。すべての編集操作はMarkdownファイルに即時反映され、テキストエディタとの双方向同期を保証する。

---
## 2. 用語定義

| 用語 | 定義 |
|------|------|
| ノード | マインドマップ上の一つの矩形要素。Markdownの見出し行に対応する |
| ルートノード | ツリーの頂点となるノード。H1見出しまたはファイル名 |
| 親ノード | 1つ以上の子ノードを持つノード |
| 子ノード | あるノードの直下に属するノード |
| 兄弟ノード | 同一親ノードに属し、同じ階層レベルを持つノード |
| 折りたたみ状態 | 子ノードを非表示にしているノードの状態 |
| フロントマター | Markdownファイル先頭の `---` で囲まれたYAMLブロック |

## 4. 機能要件
### US-01 マインドマップ表示

**ユーザーストーリー**  
ユーザーは、Markdownファイルを開いたときに、見出し構造をマインドマップとして表示したい。なぜなら、文書の構造を視覚的に把握したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-01-01 | `#`〜`######` の見出しをノードの階層として解釈して表示する | 必須 | H1〜H6の6段階すべてが正しく階層化されること |
| R-01-02 | ルートノードはH1見出しまたはファイル名（H1が存在しない場合）とする | 必須 | H1があればH1をルートとし、なければファイル名をルートとすること |
| R-01-03 | MarkdownビューとMindMapビューを切り替えられる | 必須 | エディタタイトルバーのボタン（またはコマンド）でマインドマップパネルを開閉できること |
| R-01-04 | 見出し間の本文（段落、コードブロック等）は構造変更時も保持される | 必須 | 移動・削除・追加後に見出し以外のテキストが失われないこと |
| R-01-05 | マインドマップはVS Codeのカラーテーマ（ダーク/ライト）に追従する | 推奨 | VS Code標準のCSS変数を使用していること |
| R-01-06 | ノードのテキストが省略（…）されている場合、マウスオーバー時にツールチップで全文を表示する | 必須 | テキストが省略されていないノードにはツールチップを表示しないこと |
| R-01-07 | ビューでの操作によりMarkdownに変更が反映されるたびに、自動でファイルを保存する | 必須 | 構造変更・名前変更・折りたたみ変更の反映後、VS Codeのダーティマーカーがなくなること |
| R-01-08 | 見出し直後に本文（body）が存在するノードに、インジケータードット（●）を表示する | 推奨 | ドットのホバーでbody内容がツールチップ表示されること。bodyが空のノードにはドットを表示しないこと |
| R-01-09 | 見出しノードの幅はラベルテキスト長に応じて動的に決定する | 必須 | キャンバスのテキスト幅計測（`measureNodeW`）に基づき、最小100px（`NODE_MIN_W`）〜最大400px（`NODE_MAX_W`）の範囲で幅を算出し、各ノードへインラインスタイルとして適用すること。固定幅（旧260px）を用いないこと。高さは46px（`NODE_H`）固定とし、ラベルは最大2行までクランプ（`-webkit-line-clamp: 2`）表示すること |

**補足説明**
- コマンドパレット: `Mind Map: マインドマップとして開く`
- ショートカット: `Ctrl+Shift+M`（Mac: `Cmd+Shift+M`）
- マインドマップはMarkdownエディタの横（`ViewColumn.Beside`）に表示される
- ノードデザイン: 高さ46px固定（`NODE_H`）。幅はラベル長に応じて 100〜400px の範囲で動的に決定する（R-01-09）。ボーダーは廃止し左端3pxのカラーアクセントバー（`::before`疑似要素）で階層を識別する
- 動的サイズの定数（`media/mindmap.js` 冒頭）: `NODE_MIN_W=100` / `NODE_MAX_W=400` / `NODE_H=46`、本文ノードは `BODY_MIN_W=80` / `BODY_MAX_W=360` / `BODY_H=42`（2行対応）。幅計測はラベルのフォントサイズ（見出し13px・本文12px）でテキスト幅を測り、パディング・チェックボックス分のオーバーヘッドを加算して算出する。本文ノードは子項目を持つ（折りたたみトグル表示）場合のみトグルボタン幅 `TOGGLE_W=19px` を追加で加算する（`measureNodeW(text, isBody, hasToggle)`）
- レイアウト間隔: 親右端→子左端 `H_GAP=20`、本文項目右端→ネスト子 `BODY_ITEM_GAP=12`、見出し兄弟の縦間隔 `V_GAP=16`、本文項目の縦間隔 `BODY_V_GAP=8`

### US-02 ノード移動（ドラッグ&ドロップ）

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードをドラッグして移動したい。なぜなら、構造を直感的に組み替えたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-02-01 | 同じ親ノード内での並び替えができる | 必須 | ドロップ先ノードの上端25%で「前に挿入」、下端25%で「後に挿入」となること |
| R-02-02 | 別の親ノードへの移動（階層変更）ができる | 必須 | ドロップ先ノードの中央50%で「子として追加」となること。レベルが自動更新されること |
| R-02-03 | 移動結果がMarkdownの見出し順・階層に即時反映される | 必須 | ドロップ完了後、マークダウンファイルの内容が更新されること |
| R-02-04 | ルートノードはドラッグできない | 必須 | ルートノードのドラッグ操作を無視すること |
| R-02-05 | ノードを自身の子孫ノードへドロップできない | 必須 | 循環参照が発生しないこと |
| R-02-06 | ドラッグ中はゴーストプレビューを表示する | 推奨 | ドラッグ中にノードのテキストを示す半透明ゴーストが表示されること |
| R-02-07 | ドロップ位置を水平ラインで視覚的に示す | 必須 | before/after はターゲットノードの上下辺に青ラインが表示され、inside はターゲットがハイライトされること |
| R-02-08 | H6ノードへの「子として追加」ドロップを禁止する | 必須 | H6ノードの中央領域でドラッグ中は `not-allowed` カーソルを表示しドロップを無効化すること |
| R-02-09 | ドラッグ＆ドロップのヒット判定はノード矩形の左右いっぱい＋マージンで効く | 必須 | ドロップ先判定の距離はノード中心ではなく矩形上の最近点までの距離（`distToRect`）で測り、ノード幅（動的サイズ `_w`）に追従して左右いっぱい＋許容マージン（`DROP_TOLERANCE=40`、レイアウト座標基準）まで判定が効くこと。中央付近のみに限定されないこと。縦方向の before/after/inside 判定（上端25%/下端25%/中央50%、R-02-01／R-02-02）は不変であること |

### US-03 ノード編集（インライン編集）

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードのテキストを直接編集したい。なぜなら、ビューを切り替えずに内容を修正したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-03-01 | ノードをダブルクリックでインライン編集できる | 必須 | ダブルクリックでテキスト入力フィールドがノード内に表示されること |
| R-03-02 | 編集内容がMarkdownの見出しテキストに即時反映される | 必須 | Enterキーまたはフォーカスアウト時にMarkdownが更新されること |
| R-03-03 | Escapeキーで編集をキャンセルできる | 必須 | Escapeキー押下で変更を破棄し元のテキストに戻ること |
| R-03-04 | 空文字列への変更は無効とする | 必須 | 空文字でEnterを押した場合は変更を適用しないこと |
| R-03-05 | ルートノード（ファイル名）はインライン編集・リネーム不可とする | 必須 | ルートノードのダブルクリック・`F2`・`Enter`（R-12-04 のルート編集フォールバック）いずれでもインライン編集が開始されないこと。`beginEdit` の単一チョークポイントで対象がルート（ツリー頂点）の場合は編集を開始しない。子の見出しノード・本文ノードの編集は従来どおり可能であること。R-02-04（ドラッグ不可）・R-05-04（削除不可）と同列のルート保護として扱う |

### US-04 ノード追加

**ユーザーストーリー**  
ユーザーは、マインドマップ上で新しいノードを追加したい。なぜなら、ビューを切り替えずに構造を拡張したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-04-01 | 任意のノードに子ノードを追加できる | 必須 | 右クリックメニュー「子ノードを追加」で親より1段深い子が追加されること |
| R-04-02 | 同階層に兄弟ノードを追加できる | 必須 | 右クリックメニュー「兄弟ノードを追加」で同レベルのノードが直後に追加されること |
| R-04-03 | 追加したノードがMarkdownに見出しとして即時反映される | 必須 | 追加操作後にMarkdownファイルが更新されること |
| R-04-04 | 追加されたノードのデフォルトテキストは「新しいノード」とする | 必須 | 追加直後にインライン編集モードが自動的に開始されること |

### US-05 ノード削除

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードを削除したい。なぜなら、不要な構造をビュー上から直接取り除きたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-05-01 | ノードを選択してDeleteキーまたは右クリックメニューで削除できる | 必須 | 選択状態のノードにDeleteキーが有効なこと |
| R-05-02 | 子ノードが存在する場合は確認ダイアログを表示する | 必須 | 子を持つノード削除時に「子ノードをすべて削除しますか？」確認が表示されること |
| R-05-03 | 削除結果がMarkdownに即時反映される | 必須 | 削除確定後にMarkdownファイルが更新されること |
| R-05-04 | ルートノードは削除できない | 必須 | ルートノードへの削除操作を無視すること |

### US-06 折りたたみ

**ユーザーストーリー**  
ユーザーは、ノードを折りたたんで子階層を隠したい。なぜなら、大きな文書でも全体像を把握しやすくしたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-06-01 | 子ノードを持つノードの折りたたみボタン（▼/▶）をクリックで展開/収縮できる | 必須 | クリック後に子ノードの表示/非表示が切り替わること |
| R-06-02 | 折りたたみ状態をMarkdownのフロントマター（YAML）に保存する | 必須 | 折りたたみ操作後にフロントマターが更新されること |
| R-06-03 | 折りたたまれているノードのパスをリスト形式で記録する | 必須 | フロントマターの `mindmap-collapse` キーに、ルート（ファイル名）を除いたルート直下からの相対パス（「親/子」形式）が保存されること。これによりファイルをリネームしても折りたたみ状態が保持される |
| R-06-04 | 次回ファイルを開いたとき、フロントマターを読み取り折りたたみ状態を復元する | 必須 | ファイルを再度開いた際に折りたたみ状態が復元されること。後方互換のため、旧形式（ファイル名プレフィックス付き「ルート/親/子」）のパスも復元でき、一度保存すると新形式（相対パス）へ移行する |
| R-06-05 | ツールバーの「展開」「折畳」ボタンで選択中のノードを展開/折りたたむ | 必須 | 選択ノードが折りたたみ可能な状態のときのみ操作が反映されること |
| R-06-06 | 有効でない操作（子なし・すでに展開済み・未選択等）の場合は何もしない | 必須 | 無効な状態でボタンを押しても副作用が発生しないこと |

**フロントマター形式**
```yaml
---
mindmap-collapse:
  - "ルートノード名/親ノード名"
  - "ルートノード名/別の親ノード名/孫ノード名"
---
```

### US-07 パン・ズーム

**ユーザーストーリー**  
ユーザーは、マインドマップ上でビューを移動・拡大縮小したい。なぜなら、大きな構造でも全体を把握しながら作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-07-01 | 背景ドラッグでパン（移動）できる | 必須 | ノード以外の領域（背景・接続線を含む）をドラッグしてビューが移動すること |
| R-07-02 | マウスホイールでズームできる | 必須 | ホイール操作でカーソル位置を中心にズームすること |
| R-07-03 | ツールバーの「フィット」ボタンでビュー全体を画面に収める | 必須 | ツリー全体が画面内に収まる倍率・位置に自動調整されること |
| R-07-04 | ショートカットキー `F` でフィット、`Ctrl++/Ctrl+-` でズームできる | 推奨 | キーボードショートカットが動作すること |

**補足説明**
- ノード上でのドラッグはノード移動（D&D）として扱われ、パンとは競合しない
- 接続線（ベジェ曲線）上のドラッグもパンとして機能する

### US-08 ノード上下入れ替え

**ユーザーストーリー**  
ユーザーは、同一親ノードの子ノードの並び順を上下に入れ替えたい。なぜなら、ドラッグより素早く順序を調整したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-08-01 | 右クリックメニューの「↑ 上へ移動」「↓ 下へ移動」で兄弟間の順序を変更できる | 必須 | 選択ノードが1つ上/下の兄弟と入れ替わること |
| R-08-02 | `Alt+↑` / `Alt+↓` ショートカットで選択中ノードを上下に移動できる | 必須 | キーボードショートカットで順序変更が動作すること |
| R-08-03 | 先頭ノードの「上へ移動」、末尾ノードの「下へ移動」は無効化する | 必須 | 右クリックメニューの該当項目がグレーアウト（操作不可）になること |
| R-08-04 | 移動結果がMarkdownの見出し順に即時反映される | 必須 | 順序変更後にMarkdownファイルが更新されること |

### US-09 ファイル保存（Ctrl+S）

**ユーザーストーリー**  
ユーザーは、マインドマップビューにフォーカスしたまま `Ctrl+S` で対応するMarkdownファイルを保存したい。なぜなら、他のVS Codeエディタと同じ操作感で作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-09-01 | マインドマップビュー上で `Ctrl+S`（Mac: `Cmd+S`）を押すと対応するMarkdownファイルを保存する | 必須 | キー押下後、VS CodeのタブからダーティマーカーがなくなりMarkdownが保存されること |
| R-09-02 | 保存はマインドマップ経由の最新編集内容を含む | 必須 | 直前のマインドマップ操作が反映された状態でファイルが保存されること |
| R-09-03 | 保存完了時（自動保存・手動保存いずれも）、ツールバーに「✓ 保存済」インジケーターを約1.8秒表示する | 推奨 | 保存のたびにフェードイン→フェードアウトすること。連続保存時はタイマーがリセットされること |

### US-10 Undo

**ユーザーストーリー**  
ユーザーは、マインドマップ上の操作を `Ctrl+Z` で取り消したい。なぜなら、誤操作を素早く戻せるようにしたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-10-01 | Webview内に最大50操作のUndoスタックを保持する | 必須 | 51操作目以降は最古のエントリーが破棄されること |
| R-10-02 | 各変更操作（D&D・名前変更・追加・削除・上下移動・折りたたみ変更）の直前にツリースナップショットをUndoスタックに積む | 必須 | 1回の `Ctrl+Z` で1操作分だけ戻ること |
| R-10-03 | `Ctrl+Z` でスタックから直前のスナップショットを取り出し、ツリーを復元してMarkdownに同期する | 必須 | Undo後にMarkdownファイルの内容がスナップショット時点の状態になること |
| R-10-04 | スナップショットにはノードのレイアウト座標（`_x`, `_y`, `_sh`）を含めない | 必須 | `id / text / level / collapsed / body / children` の論理データのみを保存すること |
| R-10-05 | Undo操作後は選択状態をリセットする | 推奨 | Undo実行後に選択ノードが存在しない状態になること |

### US-11 双方向同期の競合管理

**ユーザーストーリー**  
ユーザーは、マインドマップ操作中に別タブでMarkdownを直接編集した場合でも、操作後に正しい状態が表示されることを期待する。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-11-01 | 構造変更・名前変更操作中（`isOperating = true`）はMarkdown変更イベントを無視する | 必須 | D&Dやインライン編集の確定処理中にMarkdown変更イベントが割り込んでもマインドマップが中断されないこと |
| R-11-02 | 操作完了後、MarkdownをソースオブトゥルースとしてマインドマップをReparse・再描画する | 必須 | 操作完了後、Markdownと同期した最新の状態がマインドマップに反映されること |
| R-11-03 | 競合発生時はMarkdownの内容を正として扱う | 必須 | 同時編集が発生した場合、Markdownエディタ側の変更が最終的にマインドマップに反映されること |

**ユーザーストーリー（複数人同時編集での変更消失防止）**  
ユーザーは、共有ドライブやGit経由で同じ `.md` を複数人が開いて編集する状況でも、自分のマインドマップ操作が他者の変更を黙って上書きして消してしまわないことを期待する。なぜなら、リアルタイム共同編集はなくとも、知らないうちに他者の作業が失われるのは致命的だから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-11-04 | マインドマップのキャッシュツリーが「どのドキュメント内容（テキスト）からパースされたか」を base スナップショットとして記録する | 必須 | `syncFromDocument` でツリーを更新するたびに、その時点のドキュメントテキスト（改行は LF に正規化）が base として保持されること。base は書き込み成功後にも書き込んだ内容で更新されること |
| R-11-05 | 全文置換による書き込みの直前に、ライブのドキュメント内容およびディスク実体を base と比較してコンフリクトを検知する | 必須 | base と一致する場合のみ従来どおり書き込み＋保存する。base と不一致（他者が並行変更）の場合は黙って上書きしないこと。改行コードのみの差異（CRLF/LF）はコンフリクトとして扱わないこと。自分の書き込みエコー（書き込もうとしている内容と現状が一致）はコンフリクトとして扱わないこと。共有ドライブで `TextDocument` がディスクより古い場合に備え、`vscode.workspace.fs.readFile` によるディスク読み取りも判定に併用すること（自動テスト `conflictDetection.test.ts` で判定ロジックを検証） |
| R-11-06 | コンフリクト検知時はモーダル警告（`showWarningMessage`）でユーザーに解決方法を選択させる | 必須 | 「最新を読み込む（自分の編集は破棄）」選択時は他者の変更を保持してディスクから再同期し、自分の書き込みを中止すること。「自分の変更で上書き（他者の変更は破棄）」選択時のみ従来どおり全文置換すること。ダイアログを閉じる（未選択）場合は安全側（最新を読み込む）として扱うこと |
| R-11-07 | コンフリクト解決で破棄される側の内容を、衝突ファイルとしてバックアップ退避する | 推奨 | 上書き選択時は他者の版を、最新読み込み選択時は自分の版を、`<ファイル名>.conflict-<mine\|remote>-<timestamp>.md` として同じディレクトリに保存すること。バックアップ失敗はコンフリクト解決をブロックしないこと（ベストエフォート） |
| R-11-08 | 操作中（`isOperating = true`）に届いた外部変更も、操作完了後の書き込み前検知で必ずコンフリクトとして扱う | 必須 | 操作中は外部変更イベントを無視する（R-11-01）が、その間 base は更新されないため、操作完了後の全文置換前に base 比較に乗り、外部変更が検知・分岐されること。既存の `_editQueue` 直列化・echo ループガード（`applyingEdit`）が壊れないこと |

### US-12 キーボードナビゲーション

**ユーザーストーリー**  
ユーザーは、マインドマップをマウスに頼らずキーボードだけで操作したい。なぜなら、素早い構造把握・編集をキーボードで完結させたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-12-01 | `↑/↓` キーで可視ノードを順に移動（深さ優先順）できる | 必須 | 選択ノードが前後の可視ノードに切り替わること |
| R-12-02 | `→` キーで最初の子ノードに移動する。折りたたみ中なら展開してから移動する | 必須 | 子ノードがない場合は何もしないこと |
| R-12-03 | `←` キーで展開中のノードを折りたたむ。すでに折りたたみ済み（または子なし）なら親ノードに移動する | 必須 | ルートノードに親がない場合は何もしないこと |
| R-12-04 | `Enter` キーで選択中ノードの直後に兄弟ノードを追加し、直ちにインライン編集を開始する | 必須 | ルートノード（親なし）の場合は兄弟追加を行わない。ルートはインライン編集不可（R-03-05）のため、フォールバックの編集開始も行われず何も起きないこと |
| R-12-04b | `F2` キーまたはダブルクリックで選択中ノードのインライン編集を開始する | 必須 | 選択ノードにテキスト入力フィールドが表示されること |
| R-12-05 | `Tab` キーで選択中ノードに子ノードを追加し、直ちにインライン編集を開始する | 必須 | H6ノードでは `Tab` を無効化すること |
| R-12-06 | `Escape` キー（非編集中）で選択を解除する | 必須 | 選択ハイライトが消えること |
| R-12-07 | `Delete` キーで選択中ノードを削除する（既存 R-05-01 のキーボード操作と統合） | 必須 | 子を持つ場合は確認ダイアログを表示すること |
| R-12-08 | キーボード選択変更時に対象ノードが画面外にある場合、ビューをスクロールして表示する | 必須 | ノードが60px以上画面端に近い場合に自動パンすること |
| R-12-09 | 編集中（`editingId` が設定中）はナビゲーション系ショートカットを無効化する | 必須 | 入力フィールド内での操作がナビゲーションと競合しないこと |
| R-12-10 | `Ctrl+Z` でUndoスタックから直前の操作を取り消す（US-10 参照） | 必須 | Undoスタックが空の場合は何もしないこと |

**ショートカット一覧**

| キー | 動作 |
|------|------|
| `↑` / `↓` | 前後の可視ノードを選択 |
| `→` | 最初の子ノードを選択（折りたたみ中なら展開） |
| `←` | ノードを折りたたむ、または親ノードを選択 |
| `Enter` | 兄弟ノードを直後に追加して編集開始（ルートは何もしない） |
| `F2` | 選択ノードのインライン編集を開始 |
| `Tab` | 子ノードを追加して編集開始 |
| `Escape` | 選択を解除 |
| `Delete` | 選択ノードを削除 |
| `Alt+↑` / `Alt+↓` | 兄弟間の上下入れ替え |
| `Ctrl+Z` | 直前の操作を取り消す |

### US-13 本文項目のマインドマップツリー表示

**ユーザーストーリー**  
ユーザーは、ノードに紐付く本文のリスト項目（チェックリスト・箇条書き）を、マインドマップのツリー上でそのままチェック・編集したい。なぜなら、タスクや補足情報を見出しと同じビューで管理したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-13-01 | 本文中の `- [ ] item` / `- [x] item` / `- item` を、見出しノードの右に子要素として表示する（本文ノード） | 必須 | 本文ノードは見出しノードと明確に異なるデザイン（破線ボーダー・半透明背景・小フォント）で表示されること。本文ノードの幅はラベル長に応じて動的に決定され、子項目を持つ（折りたたみトグル ▼/▶ が表示される）本文ノードではトグルボタンの占有幅も加味してテキスト幅を確保し、短いテキストが不要に折り返されないこと |
| R-13-02 | 本文ノードの接続線を破線（dashed）・グレー色にして見出し間の実線と区別する | 必須 | 見出しノード→見出しノードの接続線は変更しないこと |
| R-13-03 | `- [ ] item` / `- [x] item` 行はチェックボックス付きで表示し、クリックで即座にチェック状態を切り替えてMarkdownに保存する | 必須 | `- [x]` 行はラベルに取り消し線を表示し、ノード全体を半透明にすること |
| R-13-04 | `- item`（チェックボックスなし）行はダッシュ（–）付きで表示する | 推奨 | チェックボックス行と視覚的に区別できること |
| R-13-05 | 本文ノードをダブルクリックまたは F2 でインライン編集できる。入力はテキストのみとし、`- [ ] ` プレフィックスは自動付与される | 必須 | 編集確定後に Markdown の該当行が更新されること。入力欄にチェックボックス記号を入力させないこと |
| R-13-06 | 本文ノードを選択した状態で Delete キーを押すと、その本文行を削除する | 必須 | 削除後に Markdown ファイルが更新されること |
| R-13-07 | 見出しの折りたたみボタン（▼/▶）は、本文ノードが存在する場合にも表示される | 必須 | 折りたたむと本文ノードも非表示になること |
| R-13-08 | 非リスト形式の本文行（段落・コードブロック等）はツリーに表示せず、見出しノード上のドット（●）インジケーターで存在を示す | 推奨 | ドットのホバーで本文全文がツールチップ表示されること |
| R-13-09 | 本文ノードを選択した状態で Enter キーを押すと、同一親・同一インデントレベルの兄弟本文項目をサブツリー末尾の直後に追加し、自動編集モードに入る | 必須 | 追加直後にインライン編集が開始されること。右クリックメニュー「同階層に追加」でも同等の操作が可能なこと |
| R-13-10 | 本文ノードはドラッグ&ドロップで移動できる | 必須 | 同一親内での並び替え、別の見出しノードまたは別親の本文ノード前後へのドロップが可能なこと。ドロップ位置は水平ラインで視覚的に示すこと。ドラッグ時は子項目も一緒に移動されること。移動後の階層（インデント）に応じて、トップレベル（indent=0）はチェックボックス形式、ネスト（indent>0）はダッシュ形式に自動変換されること。ドロップ先判定のヒット範囲は本文ノード矩形の左右いっぱい＋マージンで効き（R-02-09 と同方式）、中央付近のみに限定されないこと |

**補足説明**
- 本文ノードの高さは 42px（`BODY_H`・2行表示対応。見出しノード 46px よりわずかに小さい）。ラベルは最大2行までクランプ（`-webkit-line-clamp: 2`）表示する
- 本文ノードの幅はラベル長に応じて動的に決定する（最小80px=`BODY_MIN_W`〜最大360px=`BODY_MAX_W`。R-01-09 と同方式の `measureNodeW`）
- 本文ノードはネスト（子本文ノード）を持てる（US-15 参照）
- 本文行の変更は `editBody` メッセージで拡張機能に通知する
- 本文ノードのレイアウト順：見出し子ノードが上、本文ノードが下

### US-15 本文項目の階層構造（ネスト）

**ユーザーストーリー**  
ユーザーは、本文項目をMarkdownの箇条書きインデントに準拠した階層で管理し、マインドマップツリー上でも階層関係を視覚的に確認したい。なぜなら、タスクのサブタスクや補足情報を整理する粒度が必要だから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-15-01 | 本文項目はMarkdownのインデント（2スペース単位）に準拠した階層構造を持てる | 必須 | 親項目より2スペース以上多いインデントの行は子項目としてツリーに表示されること。子本文ノードは親本文ノードの実幅（`item._w`・動的）に固定ギャップ `BODY_ITEM_GAP=12px` を加えた位置（`x + item._w + BODY_ITEM_GAP`）に配置し、親子間の重なりを防ぐこと。子項目を持つ親本文ノードの実幅は、折りたたみトグルボタンの占有幅（`TOGGLE_W=19px`）をオーバーヘッドに加算して算出し、トグル表示時もテキストが折り返されないこと |
| R-15-02 | 子階層（インデントあり）の本文項目はチェックボックスを持たず、ダッシュ記号（–）で表示する | 必須 | 子本文項目を新規追加した場合も `- text`（チェックボックスなし）形式で保存されること |
| R-15-03 | Tab キーで選択中の本文項目に子項目を追加し、インライン編集を開始する | 必須 | 挿入される行のインデントが親より2スペース多いこと。追加直後にインライン編集が開始されること |
| R-15-04 | 右クリックメニュー「子項目を追加」で子本文項目を追加できる | 必須 | Tab キーと同等の動作をすること |
| R-15-05 | 本文項目に子項目が存在する場合、折りたたみボタン（▼/▶）を表示し、子項目を折りたたみ可能にする | 必須 | 折りたたむと子項目が非表示になること。折りたたみ状態はMarkdownファイルのフロントマター（`body-item-collapse:` キー）に `見出し相対パス::項目チェーン` 形式（見出し部はルート＝ファイル名を除いた相対パス）で保存し、ファイルを閉じて再度開いた際も復元されること。後方互換のため旧形式（ファイル名プレフィックス付き見出しパス）も復元でき、保存時に新形式へ移行する |
| R-15-06 | 本文項目→子本文項目の接続線は、見出し→本文項目と同じ破線グレースタイルで表示する | 推奨 | 見出し間の実線と視覚的に区別できること |

**補足説明**
- 子本文項目は親本文項目の実幅 + `BODY_ITEM_GAP`（12px）の水平位置に配置される（旧 `BODY_H_SPACE` 固定値 220px は廃止し、動的サイズに合わせた配置へ変更済み。R-15-01 参照）
- 既存Markdownに `- [x] ` 形式の子項目がある場合も子項目扱い（ただし表示はチェックボックスなし）
- 子本文項目のさらに下の階層も同様のルールで再帰的にサポートする

### US-16 チェックボックス進捗ウィジェット

**ユーザーストーリー**  
ユーザーは、ドキュメント全体のチェックボックスの完了状況を一目で把握したい。なぜなら、タスク管理の進捗を確認しながら作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-16-01 | ツールバーに全ノードのチェックボックス総数と完了数を「✓ 完了数 / 総数」形式で表示する | 必須 | チェックボックスが1つもない場合は非表示にすること。チェック状態の変更後に即時更新されること。4桁の数値（最大9999件）まで見切れずに表示できる最小幅（92px）を確保すること |
| R-16-02 | 進捗ウィジェットの背景にグリーンの進捗バーを半透明で表示する | 推奨 | 進捗率（完了数/総数）に応じてバー幅が変化すること。状態変化時にアニメーションすること |
| R-16-03 | 全チェックボックスが完了した場合、ウィジェットの枠線をグリーンに変化させる | 推奨 | 全完了→未完了に戻った際も枠線が元の色に戻ること |
| R-16-04 | キーボードショートカット一覧はツールバー常時表示をやめ、「?」ボタンのクリックでポップアップとして表示する | 必須 | ポップアップはツールバー右端に固定表示され、ボタン再クリック・領域外クリックで閉じること。ショートカット一覧は見出しノード・本文項目・共通の3セクションに分類して表示すること |

**補足説明**
- 集計対象はツリー全体のすべての見出しノードの `body` フィールド内のチェックボックス行
- ウィジェットはツールバーの折畳ボタン群の右に配置する（常時ヒントテキストは廃止）

### US-17 コピー・カット＆ペースト（Ctrl+C / Ctrl+X / Ctrl+V）

**ユーザーストーリー**  
ユーザーは、選択中のノードまたは本文項目を `Ctrl+C` でコピー・`Ctrl+X` でカットし、`Ctrl+V` で選択中ノードの子階層に貼り付けたい。なぜなら、構造を素早く複製・移動・再利用したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-17-01 | `Ctrl+C`（Mac: `Cmd+C`）で選択中の見出しノードを内部クリップボードにコピーする | 必須 | 子ノード・本文を含む全ツリー構造がコピーされること |
| R-17-02 | `Ctrl+V`（Mac: `Cmd+V`）で見出しノードを選択中ノードの**子として**貼り付ける | 必須 | 選択ノードの子階層末尾に追加されること。貼り付け先の子レベルに合わせてレベルが自動調整されること。新しいIDが付番されること。選択ノードがH6の場合は貼り付けを行わないこと |
| R-17-03 | `Ctrl+C` で選択中の本文項目を子項目ごとコピーする | 必須 | 子項目（ネスト構造）も含めて全行がコピーされること |
| R-17-04 | `Ctrl+V` で本文項目を選択中本文項目のサブツリー末尾の直後に貼り付ける | 必須 | 貼り付け先の階層に応じてチェックボックス/ダッシュ形式が自動調整されること |
| R-17-05 | `Ctrl+V` で本文項目クリップボードを見出しノード選択中に貼り付けるとトップレベル本文として追加される | 必須 | チェックボックス形式（`- [ ] text`）で末尾に追加されること |
| R-17-06 | インライン編集中（`editingId` が設定中）は Ctrl+C / Ctrl+X / Ctrl+V のマインドマップ操作を無効化する | 必須 | 入力フィールド内でのテキストコピーペーストが正常に動作すること |
| R-17-07 | `Ctrl+X`（Mac: `Cmd+X`）で選択中の見出しノードをカットする（コピー後に元ノードを削除） | 必須 | カット後に元のノードがツリーから削除されること |
| R-17-08 | `Ctrl+X` で選択中の本文項目をカットする（コピー後に元行を削除） | 必須 | カット後に元の本文行がMarkdownから削除されること |

**ショートカット追記**

| キー | 動作 |
|------|------|
| `Ctrl+C` | 選択中のノードまたは本文項目（子階層含む）をコピー |
| `Ctrl+X` | 選択中のノードまたは本文項目をカット（コピー＋削除） |
| `Ctrl+V` | コピーした内容を選択中ノードの子階層に貼り付け |

### US-18 複数選択

**ユーザーストーリー**  
ユーザーは、同じ階層のノードや本文項目を複数選択して、まとめて移動・コピー・カット・削除したい。なぜなら、複数の要素を効率よく操作したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-18-01 | `Ctrl+クリック`（Mac: `Cmd+クリック`）で見出しノードを複数選択できる | 必須 | 既に選択中のノードをCtrl+クリックすると選択解除されること。異なる階層のノードも選択可能なこと |
| R-18-02 | `Ctrl+クリック` で本文項目を複数選択できる | 必須 | 既に選択中の本文項目をCtrl+クリックすると選択解除されること |
| R-18-03 | 複数選択状態のノードをドラッグ＆ドロップで移動できる | 必須 | 選択中のいずれかのノードをドラッグすると、複数ノードがまとめて移動されること。移動先のターゲットノードの子孫である選択ノードはドロップ対象から除外されること |
| R-18-04 | 複数選択状態で `Ctrl+C` を押すと、選択中の全ノードをコピーする | 必須 | ペースト先ノードの子階層にまとめて追加されること |
| R-18-05 | 複数選択状態で `Ctrl+X` を押すと、選択中の全ノードをカットする | 必須 | カット後に元のノードがすべてツリーから削除されること |
| R-18-06 | 複数選択状態で `Delete` を押すと、選択中の全ノードを削除する | 必須 | 子ノードを持つノードが含まれる場合は確認ダイアログを表示すること |
| R-18-07 | 右クリック・矢印キー・F2などの単一操作キーは複数選択を解除して単一選択として動作する | 必須 | 右クリック時・キーナビゲーション時に複数選択状態がリセットされること |
| R-18-08 | `Escape` キーで複数選択状態を解除する | 必須 | 複数選択がすべてクリアされること |

## 5. 非機能要件
### 5.1 パフォーマンス

| ID | 要件 |
|----|------|
| NF-01 | 100ノード以下のマインドマップで描画遅延が体感されないこと（60fps相当） |
| NF-02 | Markdownへの反映は編集操作から500ms以内に完了すること |
### 5.2 信頼性

| ID | 要件 |
|----|------|
| NF-03 | マインドマップからの編集操作によりMarkdown本文（見出し以外）が失われないこと |
| NF-04 | 拡張機能のエラーがVS Code本体の動作に影響しないこと |
### 5.3 セキュリティ

| ID | 要件 |
|----|------|
| NF-05 | Webviewのコンテンツセキュリティポリシー（CSP）を適切に設定すること |
| NF-06 | nonceベースのスクリプト実行制限を実装すること |

### メッセージプロトコル

| 方向 | タイプ | ペイロード | 用途 |
|------|--------|-----------|------|
| 拡張機能 → Webview | `update` | `{ root: MindMapNode }` | ツリー更新 |
| 拡張機能 → Webview | `saved` | `{}` | ファイル保存完了通知。Webviewはインジケーターを表示する |
| Webview → 拡張機能 | `ready` | `{}` | Webview初期化完了通知。拡張機能はこれを受信してから初回ツリーを送信する |
| Webview → 拡張機能 | `structuralEdit` | `{ root: MindMapNode }` | 構造変更（移動・追加・削除・Undo復元） |
| Webview → 拡張機能 | `renameNode` | `{ id, newText }` | ノード名変更 |
| Webview → 拡張機能 | `saveCollapseState` | `{ collapsedPaths: string[] }` | 折りたたみ状態保存 |
| Webview → 拡張機能 | `save` | `{}` | ファイル保存要求（Ctrl+S / Cmd+S） |
| Webview → 拡張機能 | `editBody` | `{ id, body }` | ノードの本文のみを更新（チェックボックストグル・本文編集確定） |
| Webview → 拡張機能 | `saveBodyItemCollapseState` | `{ paths: string[] }` | 本文項目の折りたたみ状態をフロントマター（`body-item-collapse:`）へ保存 |

### ファイル構成

```
src/                       # TypeScript ソース（Extension Host 側）
  extension.ts             # コマンド登録・アクティベーション
  mindmapPanel.ts          # WebviewPanel 管理・双方向同期・自動保存
  markdownParser.ts        # Markdown → MindMapNode（フロントマター解析含む）
  markdownSerializer.ts    # MindMapNode → Markdown（折りたたみパス書き戻し含む）
  conflictDetection.ts     # 楽観的同時実行制御の純粋ロジック（base比較・改行正規化・テスト対象）
  types.ts                 # MindMapNode・メッセージ型定義
  bodyItems.ts             # 本文項目の純粋ロジック（mindmap.js と同期・テスト対象）
media/                     # Webview アセット
  mindmap.js               # レイアウト計算・描画・操作制御（動的サイズ・D&D・編集等）
  mindmap.css              # スタイル
test/                      # ユニットテスト（node:test）
  markdownRoundTrip.test.ts   # parse→serialize の冪等性・本文/フロントマター保全
  collapsedPaths.test.ts      # 折りたたみパスの抽出・適用ロジック
  bodyItems.test.ts           # 本文項目のパース・ツリー化・indent 変換ロジック
  conflictDetection.test.ts   # コンフリクト検知（base比較・改行正規化・echo判定）ロジック
esbuild.js                 # 本体ビルド（dist/extension.js を生成）
esbuild.test.js            # test/*.test.ts を dist-test/ へトランスパイル（pretest）
```

- ビルド: `npm run build`（esbuild → `dist/extension.js`）
- テスト: `npm test`（`pretest` で `esbuild.test.js` がテストをトランスパイルし、`node --test` で `dist-test/**/*.test.js` を実行）
- `test/` のユニットテストは parser/serializer の round-trip・折りたたみパス・本文項目ロジック（`src/bodyItems.ts`）を自動検証する。Webview 側の操作（D&D・編集・ズーム等）は対象外であり、§8 の受入テストでマニュアル検証する
- 本文項目ロジックは Webview（`media/mindmap.js`）が素のアセットとして配信されバンドル対象外のため、`src/bodyItems.ts` に同一ロジックを複製してテストする。両者の同期が必須（mindmap.js 側にコメント明記）

## 7. データモデル

```typescript
interface MindMapNode {
  id: string;         // 一意識別子（パース時に付番）
  text: string;       // 見出しテキスト
  level: number;      // 0=ルート(ファイル名), 1=H1, 2=H2 ... 6=H6
  children: MindMapNode[];
  collapsed: boolean;
  body: string;       // この見出し直後の本文（段落・コードブロック等）
}
```
