**文書番号**: MME-REQ-001  
**バージョン**: 1.4.0  
**作成日**: 2026-06-04  
**最終更新**: 2026-06-04  
**ステータス**: 承認済み

---
## 1. 目的・スコープ

本文書は、Visual Studio Code向けマインドマップエディタ拡張機能（`vscode-mindmap-editor`）の要求仕様を定義する。

本拡張機能は、Markdownファイルの見出し構造（H1〜H6）をインタラクティブなマインドマップとして可視化・編集する手段を提供する。すべての編集操作はMarkdownファイルに即時反映され、テキストエディタとの双方向同期を保証する。

---
## 2. 用語定義

| 用語 | 定義 |
|------|------|
| ノード | マインドマップ上の一つの矩形要素。Markdownの見出し行に対応する |
| ルートノード | ツリーの頂点となるノード。H1見出しまたはファイル名 |
| 親ノード | 1つ以上の子ノードを持つノード |
| 子ノード | あるノードの直下に属するノード |
| 兄弟ノード | 同一親ノードに属し、同じ階層レベルを持つノード |
| 折りたたみ状態 | 子ノードを非表示にしているノードの状態 |
| フロントマター | Markdownファイル先頭の `---` で囲まれたYAMLブロック |

## 4. 機能要件
### US-01 マインドマップ表示

**ユーザーストーリー**  
ユーザーは、Markdownファイルを開いたときに、見出し構造をマインドマップとして表示したい。なぜなら、文書の構造を視覚的に把握したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-01-01 | `#`〜`######` の見出しをノードの階層として解釈して表示する | 必須 | H1〜H6の6段階すべてが正しく階層化されること |
| R-01-02 | ルートノードはH1見出しまたはファイル名（H1が存在しない場合）とする | 必須 | H1があればH1をルートとし、なければファイル名をルートとすること |
| R-01-03 | MarkdownビューとMindMapビューを切り替えられる | 必須 | エディタタイトルバーのボタン（またはコマンド）でマインドマップパネルを開閉できること |
| R-01-04 | 見出し間の本文（段落、コードブロック等）は構造変更時も保持される | 必須 | 移動・削除・追加後に見出し以外のテキストが失われないこと |
| R-01-05 | マインドマップはVS Codeのカラーテーマ（ダーク/ライト）に追従する | 推奨 | VS Code標準のCSS変数を使用していること |
| R-01-06 | ノードのテキストが省略（…）されている場合、マウスオーバー時にツールチップで全文を表示する | 必須 | テキストが省略されていないノードにはツールチップを表示しないこと |

**補足説明**
- コマンドパレット: `Mind Map: マインドマップとして開く`
- ショートカット: `Ctrl+Shift+M`（Mac: `Cmd+Shift+M`）
- マインドマップはMarkdownエディタの横（`ViewColumn.Beside`）に表示される

### US-02 ノード移動（ドラッグ&ドロップ）

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードをドラッグして移動したい。なぜなら、構造を直感的に組み替えたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-02-01 | 同じ親ノード内での並び替えができる | 必須 | ドロップ先ノードの上端25%で「前に挿入」、下端25%で「後に挿入」となること |
| R-02-02 | 別の親ノードへの移動（階層変更）ができる | 必須 | ドロップ先ノードの中央50%で「子として追加」となること。レベルが自動更新されること |
| R-02-03 | 移動結果がMarkdownの見出し順・階層に即時反映される | 必須 | ドロップ完了後、マークダウンファイルの内容が更新されること |
| R-02-04 | ルートノードはドラッグできない | 必須 | ルートノードのドラッグ操作を無視すること |
| R-02-05 | ノードを自身の子孫ノードへドロップできない | 必須 | 循環参照が発生しないこと |
| R-02-06 | ドラッグ中はゴーストプレビューを表示する | 推奨 | ドラッグ中にノードのテキストを示す半透明ゴーストが表示されること |
| R-02-07 | ドロップ位置を水平ラインで視覚的に示す | 必須 | before/after はターゲットノードの上下辺に青ラインが表示され、inside はターゲットがハイライトされること |
| R-02-08 | H6ノードへの「子として追加」ドロップを禁止する | 必須 | H6ノードの中央領域でドラッグ中は `not-allowed` カーソルを表示しドロップを無効化すること |

### US-03 ノード編集（インライン編集）

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードのテキストを直接編集したい。なぜなら、ビューを切り替えずに内容を修正したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-03-01 | ノードをダブルクリックでインライン編集できる | 必須 | ダブルクリックでテキスト入力フィールドがノード内に表示されること |
| R-03-02 | 編集内容がMarkdownの見出しテキストに即時反映される | 必須 | Enterキーまたはフォーカスアウト時にMarkdownが更新されること |
| R-03-03 | Escapeキーで編集をキャンセルできる | 必須 | Escapeキー押下で変更を破棄し元のテキストに戻ること |
| R-03-04 | 空文字列への変更は無効とする | 必須 | 空文字でEnterを押した場合は変更を適用しないこと |

---
### US-04 ノード追加

**ユーザーストーリー**  
ユーザーは、マインドマップ上で新しいノードを追加したい。なぜなら、ビューを切り替えずに構造を拡張したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-04-01 | 任意のノードに子ノードを追加できる | 必須 | 右クリックメニュー「子ノードを追加」で親より1段深い子が追加されること |
| R-04-02 | 同階層に兄弟ノードを追加できる | 必須 | 右クリックメニュー「兄弟ノードを追加」で同レベルのノードが直後に追加されること |
| R-04-03 | 追加したノードがMarkdownに見出しとして即時反映される | 必須 | 追加操作後にMarkdownファイルが更新されること |
| R-04-04 | 追加されたノードのデフォルトテキストは「新しいノード」とする | 必須 | 追加直後にインライン編集モードが自動的に開始されること |

---
### US-05 ノード削除

**ユーザーストーリー**  
ユーザーは、マインドマップ上でノードを削除したい。なぜなら、不要な構造をビュー上から直接取り除きたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-05-01 | ノードを選択してDeleteキーまたは右クリックメニューで削除できる | 必須 | 選択状態のノードにDeleteキーが有効なこと |
| R-05-02 | 子ノードが存在する場合は確認ダイアログを表示する | 必須 | 子を持つノード削除時に「子ノードをすべて削除しますか？」確認が表示されること |
| R-05-03 | 削除結果がMarkdownに即時反映される | 必須 | 削除確定後にMarkdownファイルが更新されること |
| R-05-04 | ルートノードは削除できない | 必須 | ルートノードへの削除操作を無視すること |

---
### US-06 折りたたみ

**ユーザーストーリー**  
ユーザーは、ノードを折りたたんで子階層を隠したい。なぜなら、大きな文書でも全体像を把握しやすくしたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-06-01 | 子ノードを持つノードの折りたたみボタン（▼/▶）をクリックで展開/収縮できる | 必須 | クリック後に子ノードの表示/非表示が切り替わること |
| R-06-02 | 折りたたみ状態をMarkdownのフロントマター（YAML）に保存する | 必須 | 折りたたみ操作後にフロントマターが更新されること |
| R-06-03 | 折りたたまれているノードのパスをリスト形式で記録する | 必須 | フロントマターの `mindmap-collapse` キーに「ルート/親/子」形式のパスが保存されること |
| R-06-04 | 次回ファイルを開いたとき、フロントマターを読み取り折りたたみ状態を復元する | 必須 | ファイルを再度開いた際に折りたたみ状態が復元されること |
| R-06-05 | ツールバーの「展開」「折畳」ボタンで選択中のノードを展開/折りたたむ | 必須 | 選択ノードが折りたたみ可能な状態のときのみ操作が反映されること |
| R-06-06 | 有効でない操作（子なし・すでに展開済み・未選択等）の場合は何もしない | 必須 | 無効な状態でボタンを押しても副作用が発生しないこと |

**フロントマター形式**
```yaml
---
mindmap-collapse:
  - "ルートノード名/親ノード名"
  - "ルートノード名/別の親ノード名/孫ノード名"
---
```

### US-07 パン・ズーム

**ユーザーストーリー**  
ユーザーは、マインドマップ上でビューを移動・拡大縮小したい。なぜなら、大きな構造でも全体を把握しながら作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-07-01 | 背景ドラッグでパン（移動）できる | 必須 | ノード以外の領域（背景・接続線を含む）をドラッグしてビューが移動すること |
| R-07-02 | マウスホイールでズームできる | 必須 | ホイール操作でカーソル位置を中心にズームすること |
| R-07-03 | ツールバーの「フィット」ボタンでビュー全体を画面に収める | 必須 | ツリー全体が画面内に収まる倍率・位置に自動調整されること |
| R-07-04 | ショートカットキー `F` でフィット、`Ctrl++/Ctrl+-` でズームできる | 推奨 | キーボードショートカットが動作すること |

**補足説明**
- ノード上でのドラッグはノード移動（D&D）として扱われ、パンとは競合しない
- 接続線（ベジェ曲線）上のドラッグもパンとして機能する

### US-08 ノード上下入れ替え

**ユーザーストーリー**  
ユーザーは、同一親ノードの子ノードの並び順を上下に入れ替えたい。なぜなら、ドラッグより素早く順序を調整したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-08-01 | 右クリックメニューの「↑ 上へ移動」「↓ 下へ移動」で兄弟間の順序を変更できる | 必須 | 選択ノードが1つ上/下の兄弟と入れ替わること |
| R-08-02 | `Alt+↑` / `Alt+↓` ショートカットで選択中ノードを上下に移動できる | 必須 | キーボードショートカットで順序変更が動作すること |
| R-08-03 | 先頭ノードの「上へ移動」、末尾ノードの「下へ移動」は無効化する | 必須 | 右クリックメニューの該当項目がグレーアウト（操作不可）になること |
| R-08-04 | 移動結果がMarkdownの見出し順に即時反映される | 必須 | 順序変更後にMarkdownファイルが更新されること |

---
### US-09 ファイル保存（Ctrl+S）

**ユーザーストーリー**  
ユーザーは、マインドマップビューにフォーカスしたまま `Ctrl+S` で対応するMarkdownファイルを保存したい。なぜなら、他のVS Codeエディタと同じ操作感で作業したいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-09-01 | マインドマップビュー上で `Ctrl+S`（Mac: `Cmd+S`）を押すと対応するMarkdownファイルを保存する | 必須 | キー押下後、VS CodeのタブからダーティマーカーがなくなりMarkdownが保存されること |
| R-09-02 | 保存はマインドマップ経由の最新編集内容を含む | 必須 | 直前のマインドマップ操作が反映された状態でファイルが保存されること |

---
### US-10 Undo / Redo

**ユーザーストーリー**  
ユーザーは、マインドマップ上の操作を `Ctrl+Z` / `Ctrl+Y` で取り消し・やり直したい。なぜなら、誤操作を素早く戻せるようにしたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-10-01 | マインドマップ経由の編集（構造変更・名前変更）はすべて `WorkspaceEdit` を通じてMarkdownに反映し、VS Code標準のUndo/Redo履歴に統合する | 必須 | マインドマップで操作後にVS Codeの `Ctrl+Z` を押すとMarkdownが操作前の状態に戻り、マインドマップも連動して再描画されること |
| R-10-02 | 各操作（D&D・名前変更・追加・削除・上下移動）は1操作 = 1 Undoエントリーとなる | 必須 | Undoを1回押すと1操作分だけ戻ること。複数操作がまとめて戻ることがないこと |
| R-10-03 | 独自のUndo実装は持たず、VS CodeのWorkspaceEdit機構に委ねる | 必須 | マインドマップ独自のUndo履歴スタックを管理しないこと |

---
### US-11 双方向同期の競合管理

**ユーザーストーリー**  
ユーザーは、マインドマップ操作中に別タブでMarkdownを直接編集した場合でも、操作後に正しい状態が表示されることを期待する。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-11-01 | 構造変更・名前変更操作中（`isOperating = true`）はMarkdown変更イベントを無視する | 必須 | D&Dやインライン編集の確定処理中にMarkdown変更イベントが割り込んでもマインドマップが中断されないこと |
| R-11-02 | 操作完了後、MarkdownをソースオブトゥルースとしてマインドマップをReparse・再描画する | 必須 | 操作完了後、Markdownと同期した最新の状態がマインドマップに反映されること |
| R-11-03 | 競合発生時はMarkdownの内容を正として扱う | 必須 | 同時編集が発生した場合、Markdownエディタ側の変更が最終的にマインドマップに反映されること |

---
### US-12 キーボードナビゲーション

**ユーザーストーリー**  
ユーザーは、マインドマップをマウスに頼らずキーボードだけで操作したい。なぜなら、素早い構造把握・編集をキーボードで完結させたいから。

| ID | 要件 | 優先度 | 受入条件 |
|----|------|--------|---------|
| R-12-01 | `↑/↓` キーで可視ノードを順に移動（深さ優先順）できる | 必須 | 選択ノードが前後の可視ノードに切り替わること |
| R-12-02 | `→` キーで最初の子ノードに移動する。折りたたみ中なら展開してから移動する | 必須 | 子ノードがない場合は何もしないこと |
| R-12-03 | `←` キーで展開中のノードを折りたたむ。すでに折りたたみ済み（または子なし）なら親ノードに移動する | 必須 | ルートノードに親がない場合は何もしないこと |
| R-12-04 | `Enter` または `F2` キーで選択中ノードのインライン編集を開始する | 必須 | 選択ノードにテキスト入力フィールドが表示されること |
| R-12-05 | `Tab` キーで選択中ノードに子ノードを追加し、直ちにインライン編集を開始する | 必須 | H6ノードでは `Tab` を無効化すること |
| R-12-06 | `Escape` キー（非編集中）で選択を解除する | 必須 | 選択ハイライトが消えること |
| R-12-07 | `Delete` キーで選択中ノードを削除する（既存 R-05-01 のキーボード操作と統合） | 必須 | 子を持つ場合は確認ダイアログを表示すること |
| R-12-08 | キーボード選択変更時に対象ノードが画面外にある場合、ビューをスクロールして表示する | 必須 | ノードが60px以上画面端に近い場合に自動パンすること |
| R-12-09 | 編集中（`editingId` が設定中）はナビゲーション系ショートカットを無効化する | 必須 | 入力フィールド内での操作がナビゲーションと競合しないこと |

**ショートカット一覧**

| キー | 動作 |
|------|------|
| `↑` / `↓` | 前後の可視ノードを選択 |
| `→` | 最初の子ノードを選択（折りたたみ中なら展開） |
| `←` | ノードを折りたたむ、または親ノードを選択 |
| `Enter` / `F2` | 選択ノードのインライン編集を開始 |
| `Tab` | 子ノードを追加して編集開始 |
| `Escape` | 選択を解除 |
| `Delete` | 選択ノードを削除 |
| `Alt+↑` / `Alt+↓` | 兄弟間の上下入れ替え（既存） |

## 5. 非機能要件
### 5.1 パフォーマンス

| ID | 要件 |
|----|------|
| NF-01 | 100ノード以下のマインドマップで描画遅延が体感されないこと（60fps相当） |
| NF-02 | Markdownへの反映は編集操作から500ms以内に完了すること |
### 5.2 信頼性

| ID | 要件 |
|----|------|
| NF-03 | マインドマップからの編集操作によりMarkdown本文（見出し以外）が失われないこと |
| NF-04 | 拡張機能のエラーがVS Code本体の動作に影響しないこと |
### 5.3 セキュリティ

| ID | 要件 |
|----|------|
| NF-05 | Webviewのコンテンツセキュリティポリシー（CSP）を適切に設定すること |
| NF-06 | nonceベースのスクリプト実行制限を実装すること |

### メッセージプロトコル

| 方向 | タイプ | ペイロード | 用途 |
|------|--------|-----------|------|
| 拡張機能 → Webview | `update` | `{ root: MindMapNode }` | ツリー更新 |

| Webview → 拡張機能 | `structuralEdit` | `{ root: MindMapNode }` | 構造変更（移動・追加・削除） |

| Webview → 拡張機能 | `saveCollapseState` | `{ collapsedPaths: string[] }` | 折りたたみ状態保存 |
| Webview → 拡張機能 | `save` | `{}` | ファイル保存要求（Ctrl+S / Cmd+S） |

## 7. データモデル

```typescript
interface MindMapNode {
```

## 8. 受入テスト基準

| テストID | シナリオ | 期待結果 |

| AT-01 | H1〜H4の階層を持つMarkdownを開く | 4段階の階層がマインドマップに表示される |

| AT-03 | ノードをダブルクリックしてテキスト編集後Enter | Markdownの見出しテキストが変わる |
| AT-04 | 右クリック→「子ノードを追加」 | 新しい見出しがMarkdownに追加される |
| AT-05 | 子を持つノードを削除→確認ダイアログでOK | 親・子ともにMarkdownから削除される |
